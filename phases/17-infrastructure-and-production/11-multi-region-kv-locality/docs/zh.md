# 多区域 LLM 服务与 KV 缓存局部性

> 轮询负载均衡对缓存 LLM 推理是有害的。没有落在持有其前缀的节点上的请求需要支付完整预填充成本——在长提示词上 P50 约 800ms，而缓存命中时约 80ms。2026 年生产模式是缓存感知路由器（Rust 编写的 vLLM Router、llm-d 路由器），它消费 KV 缓存事件并根据前缀哈希匹配路由。最新研究（GORGO）将跨区域网络延迟作为路由目标的显式项。商业"跨区域推理"产品（Bedrock 跨区域推理、GKE 多集群网关）将推理视为不透明的——它们处理可用性，而非首 token 延迟。摩根大通和梅奥诊所在 2024 年 11 月的 us-east-1 故障转移耗时约 22 分钟。灾难恢复现实：32% 的 LLM 灾难恢复失败是因为团队备份了权重，却忘记了分词器文件或量化配置。

**类型：** 学习
**编程语言：** Python（标准库，玩具前缀缓存感知路由器模拟器）
**前置知识：** Phase 17 · 04（vLLM 服务）、Phase 17 · 06（SGLang RadixAttention）
**预计时间：** 约 60 分钟

## 学习目标

- 解释为什么轮询负载均衡破坏缓存推理，并量化首 token 延迟损失。
- 绘制缓存感知路由器图：输入（KV 缓存事件）、算法（前缀哈希匹配）、决策平局时（GPU 利用率）。
- 命名 LLM 灾难恢复失败的 32% 成因（缺少分词器文件/量化配置），并给出三文件灾难恢复清单。
- 区分商业跨区域产品（Bedrock CRI、GKE 多集群网关）与 KV 感知路由的区别。

## 问题背景

你的服务在 us-east-1、us-west-2 和 eu-west-1 运行。你在前面放了一个带轮询的 ALB。生产中前缀缓存命中率降至 8%。TTFT P50 三倍。你的 vLLM 日志显示每个请求都在支付完整预填充成本。

轮询对无状态服务是最优的。LLM 推理在设计上是有状态的——KV 缓存编码了模型所见的所有内容。盲目路由就是路由到错误的缓存。

另外，你的团队有灾难恢复方案。你将模型权重跨区域备份到 S3。区域性故障发生；你尝试故障转移；副本拒绝启动。你忘记了 tokenizer.json、量化配置和 RoPE 缩放配置在另一个你没有同步的桶中。

多区域 LLM 服务是缓存问题、路由问题和灾难恢复卫生问题——不是负载均衡器问题。

## 核心概念

### 缓存感知路由

请求携带提示词到达。路由器对前缀哈希（比如前 512 个 token）；询问每个副本"你有这个前缀的缓存吗？"。副本在分配和驱逐块时，在 pub/sub 通道上发布 KV 缓存事件。路由器选择有匹配的副本，如果没有匹配则降级到基于 GPU 利用率的决策平局。

**vLLM Router**（Rust，2026 年生产栈）：订阅 `kv.cache.block_added` 事件，维护前缀哈希 → 副本索引，以 O(1) 查找路由。无匹配时降级到最短队列深度。

**llm-d 路由器**：相同模式，Kubernetes 原生。通过 ControlPlane API 发布事件。

**SGLang RadixAttention**（Phase 17 · 06）是副本内的等价物。跨副本路由严格在其上游。

### 数字

2K token 提示词上的 TTFT P50，Llama 3.3 70B FP8，H100：
- 缓存命中（同一副本，前缀驻留）：约 80ms。
- 缓存未命中（冷预填充）：约 800ms。

10 倍差距。如果你的路由器跨副本达到 60-80% 前缀缓存命中，你以 N 副本容量近似单副本性能。如果只有 10%，你近似朴素扩缩容。

### 跨区域有新约束——网络延迟

区域间 RTT：
- us-east-1 ↔ us-west-2：约 65ms。
- us-east-1 ↔ eu-west-1：约 75ms。
- us-east-1 ↔ ap-southeast-1：约 220ms。

如果路由将请求从 us-east-1 发送到 ap-southeast-1 的热前缀，节省的预填充（800 → 80ms）会被 440ms 往返时间淹没。GORGO（2026 年研究）将此明确化——联合最小化 `prefill_time + network_latency`，而非仅最小化预填充。通常答案是除了超大 MB 级前缀（预填充占主导）外，保持区域内路由。

### 商业"跨区域推理"在这里帮不了你

AWS Bedrock 跨区域推理在容量压力期间自动将请求路由到其他区域。它优化可用性，而非首 token 延迟，并将推理视为不透明的。GKE 多集群网关也一样——服务级故障转移，不感知 KV 缓存。

即使使用这些，你仍然需要应用层缓存感知路由器。它们处理"us-east-1 着火"的情况。缓存感知路由处理首 token 延迟问题。

### 灾难恢复卫生——32% 缺少文件问题

广泛引用的 2026 年统计：32% 的 LLM 灾难恢复失败发生是因为团队备份了权重，但忘记了：

- `tokenizer.json` 或 `tokenizer.model`
- 量化配置（`quantize_config.json`、AWQ 缩放、GPTQ 零点）
- 模型特定配置（RoPE 缩放、注意力掩码、聊天模板）
- 引擎配置（`vllm_config.yaml`、采样默认值、LoRA 适配器清单）

修复是三文件最低灾难恢复清单：

1. HF 模型仓库下的所有文件（权重 + 配置 + 分词器）。
2. 引擎特定服务配置。
3. 部署清单（K8s YAML、Dockerfile、依赖锁）。

加上：每季度进行一次灾难恢复演练。摩根大通 2024 年 11 月的 us-east-1 演练能在 22 分钟内恢复，正是因为方案经过了演练。

### 数据驻留是正交问题

欧盟客户 PHI 不得离开欧盟。如果你的缓存感知路由器将巴黎发起的请求发送到 us-east-1 进行前缀匹配，无论首 token 延迟增益如何，你都违反了 GDPR。在优化缓存之前，先按数据驻留边界分区路由器。

### 需要记住的数字

- 缓存命中 vs 未命中的首 token 延迟差距：约 10 倍（2K 提示词上 80ms vs 800ms）。
- 区域间 RTT 美国-欧洲：约 75ms。
- 灾难恢复失败：32% 缺少分词器/量化配置。
- 摩根大通 us-east-1 故障转移 2024 年 11 月：22 分钟（30 分钟 SLA）。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 缓存感知路由（Cache-aware routing） | "智能负载均衡" | 根据前缀哈希匹配路由到持有 KV 缓存的副本 |
| KV 缓存事件（KV-cache events） | "缓存发布订阅" | 副本发布块添加/驱逐；路由器建立索引 |
| 前缀哈希（Prefix hash） | "缓存键" | 用于路由器查找的前 N 个 token 的哈希 |
| GORGO | "跨区域路由研究" | arXiv 2602.11688；网络延迟作为显式项 |
| 跨区域推理（Cross-region inference） | "Bedrock CRI" | AWS 产品；可用性故障转移，不感知首 token 延迟 |
| 灾难恢复清单（DR manifest） | "备份列表" | 恢复所需的每个文件——不只是权重 |
| 数据驻留（Data residency） | "GDPR 边界" | 哪个区域可以看到用户数据的法律约束 |
| RTT | "往返时间" | 网络延迟；美国-欧洲 75ms，美国-亚太 220ms |
| LLM 感知负载均衡（LLM-aware LB） | "缓存命中负载均衡" | 缓存感知路由器作为产品类别 |
