# SGLang 与 RadixAttention：前缀密集工作负载

> SGLang 将 KV 缓存视为存储在基数树中的一等可复用资源。vLLM 按先来先服务（FCFS）调度请求，而 SGLang 的缓存感知调度器优先处理具有更长共享前缀的请求——实际上是深度优先基数遍历，使热分支保持在 HBM 中。在 Llama 3.1 8B 和 ShareGPT 类 1K 提示词上，SGLang 达到约 16,200 token/秒，而 vLLM 约 12,500，领先约 29%。在前缀密集 RAG 工作负载上，优势达到 6.4 倍。在语音克隆形状的工作负载上，缓存命中率超过 86%。2026 年已在 xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS 的 40 万+ GPU 上部署。陷阱是：当前缀顺序不一致时，6.4 倍的数字会消失——顺序是工程师的杠杆。

**类型：** 学习
**编程语言：** Python（标准库，玩具基数树缓存 + 缓存感知调度器）
**前置知识：** Phase 17 · 04（vLLM 服务内部原理）、Phase 14（智能体 RAG）
**预计时间：** 约 75 分钟

## 学习目标

- 绘制 RadixAttention 图：前缀如何存储在基数树中，以及 KV 块如何在根于同一分支的序列间共享。
- 解释缓存感知调度，以及为什么 FCFS 对前缀密集流量来说是错误的。
- 给定前缀缓存命中率和提示词长度分布，计算工作负载的预期加速比。
- 命名使 6.4 倍数字真实而非空谈的提示词顺序纪律。

## 问题背景

经典服务将每个请求的提示词视为不透明的。即使 5000 个 RAG 请求都以相同的 2000 token 系统提示词加相同的检索前导开头，vLLM 也会对这个 2000 token 的前缀执行 5000 次预填充。GPU 一遍又一遍地做着相同的工作。

观察：智能体和 RAG 工作负载中的提示词几乎总是共享长前缀。系统提示词、工具 schema、少样本示例、检索头、对话历史——所有这些都在请求间重复。如果你只存储一次该前缀的 KV 缓存并复用它，就无需再次预填充。

RadixAttention 正是这样做的。Token 在基数树中建立索引；每个节点拥有从根到该节点路径上 token 序列的 KV 块。新请求遍历树：任何匹配 token 的节点都复用该节点的 KV 块。预填充成本变得与"新"后缀成正比，而非全部提示词。

挑战在于调度。如果两个请求共享一个 2000 token 的前缀，而第三个只共享 200 token，你希望将两个长共享请求放在一起服务，使长前缀保持在 HBM 中。FCFS 做的恰恰相反——它先服务最先到达的请求，可能在下一个长前缀请求到来之前就驱逐了热分支。

## 核心概念

### 基数树作为 KV 索引

基数树（紧凑前缀树）存储 token 序列。每个节点拥有一个 token 范围以及为该范围计算的 KV 块。子节点将序列延伸一个或多个 token。

```
根
 |- "你是一个有帮助的助手..."  （2000 token，124 个 KV 块）
      |- "上下文：<文档 A>..."  （500 token，31 个块）
           |- "问题：Alice..."   （80 token，5 个块）
           |- "问题：Bob..."     （95 token，6 个块）
      |- "上下文：<文档 B>..."  （520 token，33 个块）
```

来了一个带系统提示词 + "上下文：<文档 A>" + "问题：Carol" 的新请求。调度器遍历：系统前缀匹配（124 个块复用），文档 A 分支匹配（31 个块复用），然后只为"问题：Carol"分配新块（4 个块）。预填充成本：4 个块的新 token。没有树时：160 个块。预填充节省约 40 倍。

### 缓存感知调度

基数树支持的复用如果缓存频繁更换就没有意义。两个关键策略：

1. **深度优先分派**。从队列中选择下一个请求时，优先选择根于当前运行集同一分支的请求。这使热分支保持固定。
2. **分支级 LRU，而非块级 LRU**。从最少使用的叶子开始驱逐整个分支，而非单个块，使缓存形状与基数形状匹配。

FCFS 两者都违反。共享 2000 token 的请求排在共享 50 token 的请求后面，然后 2000 token 分支被驱逐以接纳 50 token 的那个。

### 需要记住的基准数字

- Llama 3.1 8B、H100、ShareGPT 1K 提示词：SGLang 约 16,200 token/秒 vs vLLM 约 12,500（约 29% 领先）。
- 前缀密集 RAG（相同系统 + 相同文档，问题各异）：SGLang 最高 6.4 倍。
- 语音克隆工作负载：86.4% 前缀缓存命中率。
- SGLang 客户的生产命中率：根据提示词纪律，50-99% 不等。
- 2026 年部署在 40 万+ GPU 上。

### 顺序陷阱

6.4 倍数字依赖于一致的提示词模板顺序。如果你的客户端在某些请求中将提示词构建为 `[系统, 工具, 上下文, 历史, 问题]`，在其他请求中构建为 `[系统, 上下文, 工具, 历史, 问题]`，基数树就找不到共享前缀。人类看起来是共享前缀的内容，对基数树来说是两个截然不同的序列。

工程师的杠杆：你的提示词模板就是缓存键。固定顺序。将所有不变内容（系统、工具、schema）放在最前面。检索上下文放在其次。用户问题放在最后。不要将动态内容交织到前缀中。

研究中的真实案例：将动态内容从可缓存前缀中移出，使一个部署的缓存命中率从 7% 变为 74%，只需一次改变。

### RadixAttention 的胜出和失败场景

胜出：
- RAG（相同检索前导，问题各异）。
- 智能体（相同工具 schema，查询各异）。
- 带长系统提示词的聊天。
- 带重复前导的语音/视觉工作负载。

失败（回退到 vLLM 级别吞吐量）：
- 带唯一提示词的单次生成（代码补全、没有系统提示词的开放式聊天）。
- 每个请求都将唯一内容交织到前缀中的动态提示词。

### 为什么这是调度器问题，而非仅仅是内核问题

你可以将 KV 复用实现为内核技巧。SGLang 的洞察是，只有当调度器保持热分支在内存中时，复用才有价值。在混合负载下，朴素的"如果可用就复用"策略会频繁更换缓存。基数树索引的调度器是将内核技巧转化为 29% 生产优势的关键。

### 与 vLLM 的相互关系

两个系统不是严格的竞争关系。2026 年 vLLM 添加了前缀缓存（`--enable-prefix-caching`）和缓存感知路由器（用 Rust 编写的 vLLM Router）。差距缩小了，但没有完全消失——SGLang 的整个栈是以基数为核心设计的；vLLM 是后来嫁接上去的。对于以前缀复用为主的工作负载，SGLang 仍是默认选择。对于没有强前缀模式的通用服务，vLLM 持平或更好。

## 动手实践

`code/main.py` 实现了玩具基数树 KV 缓存加两种策略的调度器：FCFS 和缓存感知。在两种策略上运行相同的工作负载，报告前缀缓存命中率和吞吐量差值。然后运行"打乱顺序"工作负载以展示 6.4 倍的崩溃。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| RadixAttention | "SGLang 的那个东西" | KV 缓存以基数树方式索引，使共享前缀复用块 |
| 基数树（Radix tree） | "紧凑前缀树" | 每个节点拥有 token 范围及其 KV 块的树 |
| 缓存感知调度器（Cache-aware scheduler） | "热分支优先" | 优先处理共享当前驻留分支的请求的调度器 |
| 前缀缓存命中率（Prefix-cache hit rate） | "提示词有多少是免费的" | 从复用 KV 块提供的提示词 token 比例 |
| FCFS | "先来先服务" | 破坏前缀局部性的默认调度 |
| 分支级 LRU（Branch-level LRU） | "驱逐叶子" | 与基数形状匹配的驱逐策略 |
| 提示词模板顺序（Prompt template ordering） | "缓存键" | 提示词的组件顺序决定树能共享什么 |
| 系统提示词固定（System prompt pinning） | "驻留前缀" | 将不变的系统部分保持固定以避免驱逐抖动 |
