# LLM 的 FinOps——单位经济学与多租户归因

> 传统 FinOps 在 LLM 支出上失效。成本是 token 交易，不是资源运行时间。标签无法映射——API 调用是一次交易，不是一项资产。工程决策（提示词设计、上下文窗口、输出长度）就是财务决策。2026 年的操作手册在第一天就要建立三个归因维度：按用户（`user_id`）用于座位定价和扩展，按任务（`task_id` + `route`）用于产品功能成本和优先级，按租户（`tenant_id`）用于单位经济学和续约。四个 token 层——提示词、工具、记忆、响应——一个桶隐藏了支出。多租户产品的执行阶梯：按租户速率限制（预期峰值的 2-3 倍，清晰的 429 + retry-after）；每日支出上限（合同上限的 1.5-3 倍；触发速率收紧 + 告警）；消费 z-score > 4 时的终止开关（自动暂停 + 呼叫值班）。归因模式：标签聚合、遥测连接器（追踪 ID → 计费；最高精度）、采样外推、模型分配、事件溯源、实时流处理。单位指标：每个已解决查询的成本、每个生成制品的成本——不是 $/M token。事后标签总会遗漏；在请求创建时就进行埋点。

**类型：** 学习
**编程语言：** Python（标准库，带终止开关的玩具成本归因模拟器）
**前置知识：** Phase 17 · 13（可观测性）、Phase 17 · 14（缓存）
**预计时间：** 约 60 分钟

## 学习目标

- 解释为什么传统 FinOps（标签 + 层级）在 LLM 支出上失效，并说出三个新的归因维度。
- 列举四个 token 层（提示词、工具、记忆、响应），以及为什么单桶计费会隐藏成本。
- 为多租户产品设计执行阶梯（速率 → 支出上限 → 终止开关）。
- 选择单位指标（每个已解决查询/制品的成本），而非 $/M token。

## 问题背景

你的账单显示 4 万美元。你不知道：
- 哪个租户花了这些钱。
- 哪个产品功能驱动了这些支出。
- 是否有个别用户在滥用。
- 是提示词膨胀、工具调用还是记忆放大是罪魁祸首。

提供商侧的标签聚合对云资源（EC2、S3）有效，那些资源的标签会传播到明细项。LLM API 调用不会自动打标签——你必须在调用点为用户/任务/租户打上标记并贯穿始终。事后归因总会遗漏边缘情况。

## 核心概念

### 三个归因维度

**按用户**（`user_id`）：谁花了多少钱。驱动座位定价、扩展对话、识别重度用户。

**按任务**（`task_id` + `route`）：哪个产品功能花了多少钱。驱动功能优先级、关闭高成本功能的决策。

**按租户**（`tenant_id`）：哪个客户有利可图。驱动单位经济学、续约定价、层级阈值。

第一天就在调用点埋点所有三个维度。事后补救总是更糟糕。

### 四个 Token 层

| 层级 | 示例 | 占总量的典型比例 |
|------|------|----------------|
| 提示词 | 系统提示词 + 用户输入 | 40-60% |
| 工具 | 工具调用结果回填 | 20-40%（智能体工作负载） |
| 记忆 | 历史对话/检索文档 | 10-30% |
| 响应 | 模型输出 | 10-30% |

将四层都放在一个桶里会使优化变得盲目。在归因方案中将它们分开。

### 执行阶梯

1. **按租户速率限制**。预期峰值的 2-3 倍。返回带 `Retry-After` 的 429。租户感受到摩擦；没有意外账单。

2. **每日支出上限**。合同上限的 1.5-3 倍。触发：收紧速率限制 + 告警客户成功团队。

3. **消费 z-score > 4**（相对于租户基线）的终止开关。自动暂停租户；呼叫值班；上报给运维 + CS。

### 归因模式

- **标签聚合**：打元数据标头；之后聚合。简单；粗略。
- **遥测连接器**：通过追踪 ID 将追踪连接到计费。最高精度。成熟团队的做法。
- **采样外推**：采样 5-10%，乘以倍数。对粗略支出有效；错过尾部。
- **模型分配**：回归推断成本驱动因素。用于没有标签的历史数据。
- **事件溯源**：成本作为流中的事件（Kafka / Kinesis）。实时。
- **实时流处理**：仪表板亚秒级更新。

### 每 X 的成本是单位指标

$/M token 是供应商用语。产品指标：

- 每个已解决支持工单的成本。
- 每篇生成文章的成本。
- 每个成功智能体任务的成本。
- 每用户会话分钟的成本。

将成本与产品结果挂钩。否则优化就没有锚点。

### 成本归因追踪结构

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

每次调用发出。存储在数据湖中。按维度聚合。Phase 17 · 13 可观测性技术栈是它的归宿。

### 叠加节省技术栈

技术栈：缓存 + 批处理 + 路由 + 网关。四者叠加：
- L2 缓存（Phase 17 · 14）：输入约便宜 10 倍。
- 批处理（Phase 17 · 15）：折扣 50%。
- 路由到廉价模型（Phase 17 · 16）：成本降低 60%。
- 网关效率（Phase 17 · 19）：冗余 + 重试。

最佳叠加情况：约为朴素基线的 5-10%。大多数团队有 2-3 个杠杆在使用；很少有团队叠加全部四个。

### 需要记住的数字

- 归因维度：按用户、按任务、按租户。
- 四个 token 层：提示词、工具、记忆、响应。
- 终止开关：消费 z-score > 4。
- 单位指标：每个已解决查询的成本，而非 $/M token。
- 叠加优化：可能达到基线的约 5-10%。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 按用户归因（Per-user attribution） | "用户级成本" | 每次调用打上 `user_id` |
| 按任务归因（Per-task attribution） | "功能成本" | `task_id` + `route` 标识产品功能 |
| 按租户归因（Per-tenant attribution） | "客户成本" | `tenant_id`；驱动单位经济学 |
| 四个 token 层（Four token layers） | "成本层" | 提示词 + 工具 + 记忆 + 响应 |
| 速率限制（Rate limit） | "429 防护" | 在网关强制执行的按租户上限 |
| 每日支出上限（Daily spend cap） | "每日上限" | 带告警的租户范围预算 |
| 终止开关（Kill switch） | "自动暂停" | 消费 z-score > 4 触发自动暂停 |
| 每个已解决的成本（Cost per resolved） | "产品单位指标" | 与产品结果挂钩的成本，不是 token |
| 遥测连接器（Telemetry joiner） | "追踪到计费" | 最高精度的归因模式 |
| 叠加优化（Stacked optimization） | "缓存+批处理+路由+网关" | 叠加节省到约 5-10% 基线 |
