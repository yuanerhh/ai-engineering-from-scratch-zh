# 批处理 API——50% 折扣作为行业标准

> 每家主要提供商都提供带 50% 折扣和约 24 小时周转时间的异步批处理 API。OpenAI、Anthropic、Google 以及大多数推理平台（Fireworks 批处理层、Together 批处理）实现了相同的模式。将批处理与提示词缓存叠加，隔夜管道的成本降至同步未缓存成本的约 10%。规则简单得残酷：如果不是交互式的，它就属于批处理。内容生成管道、文档分类、数据提取、报告生成、批量标注、目录标签——任何能容忍 24 小时延迟的，在迁移到批处理之前都是白白浪费的钱。2026 年的生产模式是将每个新 LLM 工作负载分类到三个通道：交互式（带缓存的同步）、半交互式（带回退的异步队列）、批处理（隔夜，叠加缓存输入）。假装是交互式但能容忍分钟级延迟的工作负载浪费最多。

**类型：** 学习
**编程语言：** Python（标准库，玩具批处理 vs 同步成本模拟器）
**前置知识：** Phase 17 · 14（提示词与语义缓存）
**预计时间：** 约 45 分钟

## 学习目标

- 命名三个提供商批处理 API（OpenAI、Anthropic、Google）以及通用的 50% 折扣 + 24 小时周转保证。
- 计算在隔夜分类工作负载上叠加批处理 + 缓存输入的成本，并与同步未缓存基线对比。
- 将工作负载分类到交互式/半交互式/批处理，并说明理由。
- 命名两个陷阱：部分交互性（用户期望比 24 小时更快）和输出 schema 漂移（每个提供商的批处理文件格式不同）。

## 问题背景

你的团队发布了一个夜间报告生成管道。5 万份文档，每份摘要，聚合摘要，起草执行简报。同步运行需要 4 小时，每晚 2000 美元。你听说了批处理 API。

批处理给你 50% 折扣。你还在系统提示词（5 万次调用共享）上启用了提示词缓存。叠加后，账单降至每晚 180 美元——约为基线的 9%。相同管道，三处配置更改。

批处理是 LLM 成本工具箱中没有人去使用的最便宜的杠杆。原因主要是组织性的：团队认为"实时"，而 SLA 实际上是"到早上"。本课是关于不把 90% 的账单白白留在桌上。

## 核心概念

### 三个批处理 API

**OpenAI 批处理 API**：包含请求列表的 JSONL 文件上传。承诺 24 小时周转（实际通常约 2-8 小时）。输入和输出 token 50% 折扣。`/v1/batches` 端点。符合条件的缓存输入还可在此基础上获得缓存输入定价。

**Anthropic 消息批处理**：JSONL 上传。24 小时周转。50% 折扣。支持 `cache_control`——缓存写入是显式的，读取在批次内自动发生。

**Google Vertex AI 批量预测**：BigQuery 或 GCS 输入。Gemini 类似的 50% 折扣。与 Vertex 管道集成。

### 语义上：异步，而非缓慢

批处理是"我承诺在 24 小时内返回"——而非"这需要 24 小时"。典型 P50 是 2-6 小时。提供商在 GPU 库存利用率低的非高峰窗口调度你的批次。

### 与缓存叠加

带相同 4K token 系统提示词的 5 万份文档摘要：

- 同步未缓存：5 万 × （$输入 × 4000 + $输出 × 200）按全价。
- 同步有缓存：第一次写入后系统提示词被缓存；剩余 49999 次获得 10 倍更便宜的输入。
- 批处理有缓存：上述所有加上读写各 50% 折扣。

叠加：批处理 + 缓存 = 约同步未缓存账单的 10%。任何隔夜运行且有共享系统提示词的工作负载都应该使用这个。

### 工作负载分类

**交互式** — 用户等待响应。首 token 延迟重要。带提示词缓存的同步调用。无法批处理。

**半交互式** — 用户提交任务，几分钟后查看。带回退到同步的异步队列（如果批处理不可用）。想象中等量的 RAG 索引。

**批处理** — 用户期望"到早上"或"下一小时"得到结果。内容管道、大规模分类、离线分析。始终批处理，始终叠加缓存。

常见错误：因为管道是生产级的就把所有内容分类为交互式。生产级不是延迟规范——SLA 才是。

### 部分交互性陷阱

一些功能看起来是交互式的，但能容忍 5-10 分钟。示例：带"刷新"按钮的夜间客户健康报告。用户点击刷新；等待 10 分钟没问题。团队将其作为同步部署。50 个并发刷新的成本是批处理后通过邮件交付成本的 10 倍。

要问的问题："24 小时对这个用户意味着什么？"如果答案是"他们不会注意到"，批处理它。

### 输出 schema 陷阱

批处理文件格式因提供商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式嵌入其中。
- Vertex：BigQuery 表或带 TFRecord 的 GCS 前缀。

编写"一个批处理客户端"跨提供商意味着每个提供商都需要适配器代码。宣传多提供商批处理的网关（Portkey、LiteLLM 某些层级）仍然是对原始格式的薄封装。

### 需要记住的数字

- 各提供商批处理折扣：输入 + 输出一律 50%。
- 周转 SLA：24 小时保证，典型 P50 是 2-6 小时。
- 叠加批处理 + 缓存输入：约同步未缓存成本的 10%。
- 工作负载分类规则：如果 24 小时延迟可接受，始终批处理。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 批处理 API（Batch API） | "异步折扣" | 24 小时周转，50% 折扣 |
| JSONL | "批处理格式" | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| 消息批处理（Message Batches） | "Anthropic 批处理" | Anthropic 的批处理 API 产品名称 |
| 批量预测（Batch prediction） | "Vertex 批处理" | Vertex AI 的批处理 API 产品 |
| 周转 SLA（Turnaround SLA） | "24 小时承诺" | 保证，不是典型值；典型是 2-6 小时 |
| 工作负载分类（Workload triage） | "交互性决策" | 交互式/半交互式/批处理路由决策 |
| 输出 schema（Output schema） | "响应格式" | 每提供商的 JSONL 布局；不可移植 |
| 叠加折扣（Stacked discount） | "批处理 + 缓存" | 两者都适用时约为未缓存同步账单的 10% |
