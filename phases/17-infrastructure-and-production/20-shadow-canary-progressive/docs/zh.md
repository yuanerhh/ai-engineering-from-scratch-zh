# 影子流量、金丝雀发布与 LLM 渐进式部署

> LLM 发布结合了软件部署中最困难的部分：没有单元测试、失败模式分散、信号延迟。流程是：(1) 影子模式——将生产请求复制到候选模型，记录日志，对比结果，对用户零影响；能发现明显的分布问题，但不是质量保证；(2) 金丝雀发布——渐进流量切换 10% → 25% → 50% → 75% → 100%，每步都有卡口；跟踪延迟百分位数、每请求成本、错误/拒绝率、输出长度分布、用户反馈率；(3) 在稳定性确认后对明显不同的候选方案进行 A/B 测试。非确定性是不可消除的——由于 GPU 浮点非结合性加批量大小方差，相同输入的准确率变化最高可达 15%。成本是变量而非常量——质量提升 20% 的模型每次调用可能贵 3 倍。回滚速度是决定性因素：如果回滚需要重新部署，你的速度太慢了。策略存储在配置/标志中；模型存储在带固定摘要的注册表中；回滚 = 翻转策略 + 还原阈值 + 固定旧模型，只需几秒钟。

**类型：** 学习
**编程语言：** Python（标准库，玩具金丝雀进度模拟器）
**前置知识：** Phase 17 · 13（可观测性）、Phase 17 · 21（A/B 测试）
**预计时间：** 约 60 分钟

## 学习目标

- 区分影子模式（零影响对比）、金丝雀（实时流量渐进）和 A/B 测试（稳定性确认后的对比）。
- 列举五个 LLM 专属金丝雀指标（延迟、每请求成本、错误/拒绝、输出长度分布、用户反馈）。
- 解释为什么 LLM 非确定性（最高 15%）改变了发布中"稳定"的含义。
- 设计一个秒级回滚路径（策略翻转）而非小时级（重新部署）。

## 问题背景

你上线了一个新模型。离线评估显示准确率提升 3%。你在生产环境中切换它。24 小时内，成本上升 40%，用户负面反馈上升 8%，三个客户工单报告"奇怪的回答"。你回滚。重新部署需要 3 小时。你的周末毁了。

这一切都是可以避免的。影子模式会在任何用户看到之前发现 40% 的成本激增。金丝雀会在负面反馈上升时停在 10%。策略标志回滚只需 30 秒。这种纪律性填补了"离线评估看起来不错"和"真实用户满意"之间的差距。

## 核心概念

### 影子模式

候选模型接收与生产相同的请求；输出被记录，不返回给用户。对用户零影响。记录：

- 输出内容（与生产输出的差异）。
- Token 计数（成本差值）。
- 延迟。
- 拒绝和错误。

能发现：成本暴增、长度回归、明显的拒绝变化、硬错误。无法发现：用户能感知的质量差异。影子是冒烟测试，不是质量测试。

### 金丝雀发布

带卡口的渐进流量切换。典型进度：1% → 10% → 25% → 50% → 75% → 100%。每步在 5 个指标上设置卡口：

1. **延迟百分位数** — P50、P95、P99。触发条件：金丝雀 P99 > 基线的 1.5 倍。
2. **每请求成本** — 混合 $。触发条件：超过基线 20%。
3. **错误/拒绝率** — 5xx 加显式拒绝。触发条件：基线的 2 倍。
4. **输出长度分布** — 均值 + P99。触发条件：分布偏移。
5. **用户反馈率** — 负面反馈/工单提交。触发条件：基线的 1.5 倍。

### 非确定性是新的方差来源

相同输入产生不相同的输出。原因：

- GPU 浮点非结合性（浮点归约顺序随批次而变化）。
- 批量大小方差（同一提示词在 128 批次中与 16 批次中处理不同）。
- 采样（temperature > 0）。

实测：相同评估集上逐次运行准确率变化最高 15%。发布中"稳定"意味着指标在预期方差范围内，而不是与基线完全相同。将卡口设置在噪声上限以上。

### 成本是变量

质量提升 20% 的模型每次调用可能贵 3 倍。每请求成本是五个卡口之一。发布一个破坏单位经济的"更好"模型就是回滚案例。

### 回滚是武器

- 策略标志（功能标志系统）：在配置中翻转百分比；只需几秒钟。
- 模型固定（注册表摘要）：固定模型不自动升级。
- 回滚 = 还原标志 + 将固定摘要设为上一版本。秒级，不是小时级。

如果你的技术栈需要重新部署才能回滚，先解决这个问题再发布。

### 工具

**Argo Rollouts** / **Flagger** — Kubernetes 渐进式交付控制器。与 Istio/Linkerd 加权路由集成。

**Istio 加权路由** — 服务网格级别的流量分割。

**KServe / Seldon Core** — 内置金丝雀的模型服务。

**功能标志** — LaunchDarkly、Flagsmith、Unleash。策略级翻转，无需重新部署。

### 指标节奏

金丝雀卡口根据流量量每 5-15 分钟检查一次。1% 流量配合 10 req/min 每个窗口给出 50-150 个数据点——延迟够用，但用户反馈噪声大。10% 给出约 10 倍更多数据点。每步进度应暂停足够长的时间，在该步积累足够样本。

### A/B 测试步骤是可选的

如果新模型明显不同（不同行为、不同成本曲线、不同语气），在金丝雀通过后以 50% 进行 A/B 测试。如果只是改进版本，当金丝雀卡口通过时直接升至 100%。

### 需要记住的数字

- 金丝雀进度：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：相同输入逐次运行最高 15% 方差。
- 五个金丝雀指标：延迟、成本、错误/拒绝、输出长度、用户反馈。
- 成本卡口：超过基线 20% 即触发。
- 回滚：秒级，不是小时级。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 影子模式（Shadow mode） | "复制到新模型" | 零影响发送到候选模型用于记录 |
| 金丝雀（Canary） | "渐进流量" | 带卡口的逐步用户暴露发布 |
| 卡口（Gates） | "发布检查" | 阻止进度的指标阈值 |
| 非确定性（Non-determinism） | "LLM 方差" | 不可消除的逐次运行差异 |
| 策略标志（Policy flag） | "标志翻转回滚" | 配置级回滚，秒级而非小时级 |
| 模型固定（Model pin） | "注册表摘要" | 对模型版本的不可变引用 |
| Argo Rollouts | "K8s 渐进式" | Kubernetes 原生金丝雀/回滚控制器 |
| KServe | "推理 K8s" | 带金丝雀原语的模型服务 |
| Istio 加权路由（Istio weighted） | "网格分流" | 服务网格流量分割器 |
