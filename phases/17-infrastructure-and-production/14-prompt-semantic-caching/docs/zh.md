# 提示词缓存与语义缓存经济学

> **定价数据截至 2026-04。** 以下数字反映本课发布时从供应商费率卡获取的信息；在引用之前请对照链接文档进行核实，因为费率会定期更新。

> 缓存发生在两层。L2（提供商级）提示词/前缀缓存为重复前缀复用注意力 KV——Anthropic 的提示词缓存文档宣传长提示词最高可降低 90% 成本和 85% 延迟；对于 Claude 3.5 Sonnet，缓存读取为 0.30 美元/百万 token，而新鲜输入为 3.00 美元/百万 token，5 分钟 TTL，1 小时 TTL 选项有 2 倍写入溢价（docs.anthropic.com，2026-04）。OpenAI 提示词缓存对 ≥1024 token 的提示词自动应用，缓存输入定价约为新鲜输入的 90% 折扣（platform.openai.com，2026-04）；具体的每模型缓存费率取决于当前费率卡。L1（应用级）语义缓存在嵌入相似性命中时完全跳过 LLM。供应商声称的"95% 准确率"是指匹配正确性，而非命中率——报告的生产命中率从 10%（开放式聊天）到 70%（结构化 FAQ）不等；两家提供商都没有发布官方基线，因此请将这些视为社区遥测数据而非保证。生产陷阱：并行化会杀死缓存（在首次缓存写入完成前发出的 N 个并行请求会使支出膨胀数倍），前缀中的动态内容完全阻止缓存命中。ProjectDiscovery 报告通过将动态文本移出可缓存前缀，命中率从 7% 升至 74%（2025-11）。

**编程语言：** Python（标准库，玩具双层缓存模拟器）
**前置知识：** Phase 17 · 04（vLLM 服务内部原理）、Phase 17 · 06（SGLang RadixAttention）
**预计时间：** 约 60 分钟

## 学习目标

- 区分 L2 提示词/前缀缓存（提供商处的 KV 复用）与 L1 语义缓存（相似提示词绕过 LLM）。
- 解释 Anthropic 的 `cache_control` 显式标记和两种 TTL 选项（5 分钟 vs 1 小时）及其价格乘数。
- 给定命中率、提示词/响应混合比例和 token 价格，计算预期月度节省。
- 命名使账单膨胀 5-10 倍的并行化反模式，以及使命中率崩溃的动态内容反模式。

## 问题背景

你为 RAG 服务添加了提示词缓存。账单没有变化。你测量命中率；只有 7%。你的提示词看起来是静态的，但其实不是——系统提示词包含精确到分钟的当前日期、请求 ID 和随机重排序的示例（为了多样性）。每个请求写入新的缓存条目，读取为零。

另外，你的智能体每个用户问题运行 10 个并行工具调用。所有 10 个在第一次缓存写入完成之前到达提供商。10 次写入，0 次读取。你的账单是"有缓存"预期成本的 5-10 倍。

缓存是一种协议，不是一个标志。两层，两种不同的失败模式。

## 核心概念

### L2——提供商提示词/前缀缓存

提供商存储可缓存前缀的注意力 KV，并在下一个匹配前缀的请求上复用。你支付一次写入成本，读取几乎免费。

**Anthropic（Claude 3.5 / 3.7 / 4 系列）**：在请求中使用显式 `cache_control` 标记。你标注哪些块是可缓存的。TTL：5 分钟（写入成本为 1.25 倍基础）或 1 小时（写入成本为 2 倍基础）。缓存读取：Claude 3.5 Sonnet 上 0.30 美元/百万 vs 新鲜 3.00 美元/百万——便宜 10 倍（docs.anthropic.com，截至 2026-04）。每模型费率不同（Opus/Haiku 单独发布）；始终对照当前定价页面核实。

**OpenAI**：对 ≥1024 token 的提示词自动缓存（platform.openai.com，2026-04）。无需显式标志。在当前 gpt-4o/gpt-5 费率卡上，缓存输入约比新鲜输入便宜 10 倍。文档和发布说明都没有发布官方命中率基线；社区报告在精心提示词设计下集中在 30-60% 左右。监控 `usage.cached_tokens` 来测量你自己的情况。

**Google（Gemini）**：通过显式 API 进行上下文缓存；100 万 token 上下文意味着缓存回报更高。

**自托管（vLLM、SGLang）**：Phase 17 · 06 涵盖 RadixAttention——你自己计算资源上的相同模式。

### L1——应用级语义缓存

在调用 LLM 之前，对提示词哈希，嵌入它，并查找相似的缓存请求（余弦相似度高于阈值，通常 0.95+）。命中时返回缓存响应。未命中时调用 LLM 并缓存结果。

开源：Redis 向量相似度、GPTCache、Qdrant。商业：Portkey Cache、Helicone Cache。

供应商准确率声明是指返回的缓存响应在语义上合适的频率——而非你命中的频率。生产命中率：

- 开放式聊天：10-15%。
- 结构化 FAQ / 支持：40-70%。
- 代码问题：20-30%（小变体会杀死命中）。
- 重复提示词的语音智能体：50-80%（固定集语音规范化）。

### 并行化反模式

你的智能体并行发出 10 个工具调用。所有 10 个都有相同的 4K token 系统提示词。Anthropic 缓存写入是按请求的；第一次缓存写入在提供商看到提示词后约 300ms 完成。请求 2-10 在同一毫秒窗口内到达，每个都看到缓存未命中。你支付 10 次写入溢价，0 次读取折扣。

修复：先单独发出请求 1，然后在请求 1 的缓存填充后发出 2-10。给第一个工具调用增加 300ms；节省 5-10 倍账单。

### 动态内容反模式

你的系统提示词看起来像：

```
你是一个有帮助的助手。当前时间是 14:32:17。
用户 ID：abc123。今天是星期二...
```

每个请求都是唯一的。每个请求都写入。命中率为零。

修复：将真正静态的内容移到可缓存前缀；在缓存边界之后追加动态内容：

```
[可缓存]
你是一个有帮助的助手。[规则、示例、说明]
[/可缓存]
[动态，不缓存]
当前时间：14:32:17。用户：abc123。
```

ProjectDiscovery 通过这种方式将缓存命中率从 7% 提升到 74%，并发布了详细分析。

### 为隔夜工作负载叠加批处理 + 缓存

批处理 API（Phase 17 · 15）以 24 小时周转时间提供 50% 折扣。在此基础上缓存输入再获得约 10 倍。通过叠加，隔夜分类、标注和报告生成工作负载可以降至同步非缓存成本的约 10%。

### 需要记住的数字

以下定价点截至 2026-04 从链接的供应商文档获取，每隔几个月会有变动——在依赖之前重新核实。

- Anthropic 缓存读取：Claude 3.5 Sonnet 上 0.30 美元/百万，约比新鲜输入便宜 10 倍（docs.anthropic.com）。
- Anthropic 缓存写入溢价：1.25 倍（5 分钟 TTL）或 2 倍（1 小时 TTL）。
- OpenAI 自动缓存：适用于 ≥1024 token 的提示词；在当前费率卡上缓存输入定价约为新鲜输入的 10%（platform.openai.com）。
- 语义缓存命中率（社区报告）：开放聊天约 10%；结构化 FAQ 最高约 70%。非供应商文档基线。
- ProjectDiscovery：通过将动态内容移出前缀，命中率从 7% → 74%（项目博客，2025-11）。
- 并行化反模式：N 个并行请求错过首次缓存写入时，账单膨胀通常报告 5-10 倍。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| L2 提示词缓存（L2 prompt cache） | "前缀缓存" | 提供商为重复前缀存储 KV |
| `cache_control` | "Anthropic 缓存标记" | 标注可缓存块的显式属性 |
| 缓存写入溢价（Cache write premium） | "写入税" | 首次未命中写缓存的额外成本（1.25 倍或 2 倍） |
| L1 语义缓存（L1 semantic cache） | "嵌入缓存" | 调用 LLM 之前的应用级哈希和嵌入 |
| GPTCache | "LLM 缓存库" | 流行的 OSS L1 缓存库 |
| 缓存命中率（Cache hit rate） | "命中/总量" | 从缓存提供服务的请求比例 |
| 并行化反模式（Parallelization anti-pattern） | "N 次写入陷阱" | N 个并行请求未命中缓存 N 次 |
| 动态内容陷阱（Dynamic content trap） | "提示词中的时间陷阱" | 前缀中的动态字节破坏命中率 |
| RadixAttention | "副本内缓存" | SGLang 的前缀缓存实现 |
