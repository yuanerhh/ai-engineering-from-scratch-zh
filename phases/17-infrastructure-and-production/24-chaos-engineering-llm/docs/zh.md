# LLM 生产的混沌工程

> 2026 年，LLM 的混沌工程是一门独立学科。在生产中运行实验的前提条件：定义 SLI/SLO、追踪+指标+日志可观测性、自动回滚、Runbook、值班人员。架构有四个平面：控制（实验调度器）、目标（服务、基础设施、数据存储）、安全（护栏 + 中止 + 流量过滤）、可观测性（指标 + 追踪 + 日志），以及反馈（融入 SLO 调整）。护栏是强制性的：如果每日错误预算消耗 > 预期的 2 倍，燃烧率告警暂停实验；抑制窗口 + 追踪 ID 关联为告警噪声去重。节奏：每周小型金丝雀 + SLO 审查；每月演练 + 事后分析；每季度跨团队弹性审计 + 依赖关系映射。LLM 专属实验：内存过载、网络故障、提供商宕机、畸形提示词、KV 缓存驱逐风暴。工具：Harness 混沌工程（LLM 驱动的建议、影响半径缩减、MCP 工具集成）；LitmusChaos（CNCF）；Chaos Mesh（CNCF Kubernetes 原生）。

**类型：** 学习
**编程语言：** Python（标准库，玩具混沌实验运行器）
**前置知识：** Phase 17 · 23（AI 的 SRE）、Phase 17 · 13（可观测性）
**预计时间：** 约 60 分钟

## 学习目标

- 列举五个混沌工程前提条件（SLI/SLO、可观测性、回滚、Runbook、值班），并解释跳过任何一个会破坏整个实践。
- 绘制四个平面（控制、目标、安全、可观测性）及其反馈回路到 SLO 的图示。
- 枚举五个 LLM 专属实验（内存过载、网络故障、提供商宕机、畸形提示词、KV 驱逐风暴）。
- 根据技术栈选择工具——Harness、LitmusChaos、Chaos Mesh。

## 问题背景

传统技术栈的混沌测试已经成熟。LLM 技术栈带来了新的故障模式。一个带毒化字符的 4K token 提示词使分词器卡住 12 秒。上游提供商返回 429；你的网关重试；服务因重试放大的并发而 OOM。突发负载下的 KV 缓存驱逐风暴导致重新预填充级联，使计算资源饱和。

这些都不会出现在单元测试中。混沌工程是在用户发现这些问题之前你自己发现它们的方式。

## 核心概念

### 前提条件

在生产中运行混沌实验，必须具备：

1. **SLI/SLO** — 定义的服务级别指标和目标。
2. **可观测性** — 追踪、指标、日志，已接入仪表板。
3. **自动回滚** — Phase 17 · 20 策略标志回滚。
4. **Runbook** — 结构化，Phase 17 · 23。
5. **值班** — 有人能响应。

缺少任何一个意味着混沌会变成真实的事故。

### 四个平面 + 反馈

**控制平面** — 实验调度器（Litmus 工作流、Chaos Mesh 调度、Harness UI）。

**目标平面** — 服务、Pod、节点、负载均衡器、数据存储。

**安全平面** — 终止开关、抑制窗口、影响半径限制、错误预算卡口。

**可观测性平面** — 常规指标 + 追踪 ID 关联，用于区分混沌引发的故障和自然故障。

**反馈回路** — 发现结果反馈到 SLO 调整、Runbook 更新、代码修复。

### 护栏是强制性的

- **燃烧率告警**：如果每日错误预算消耗超过预期的 2 倍，暂停实验。
- **抑制窗口**：实验期间在影响半径内静音非实验告警。
- **追踪 ID 关联**：所有实验引发的错误都带标签，以便值班人员去重。

### 五个 LLM 专属实验

1. **内存过载** — 以高并发发送长上下文请求，强制 KV 缓存抢占风暴。观察：服务是优雅降级还是崩溃？

2. **网络故障** — 切断推理网关和提供商之间的连接。观察：故障转移是否在 SLA 内启动？（Phase 17 · 19）

3. **提供商宕机模拟** — OpenAI 100% 返回 429。观察：路由是否故障转移到 Anthropic？（Phase 17 · 16、19）

4. **畸形提示词** — 注入使分词器卡住的载荷（例如，深度嵌套的 Unicode、超大 UTF-8 码点）。观察：单个请求是否会锁定一个工作进程？

5. **KV 驱逐风暴** — 通过饱和 vLLM 块预算强制驱逐。观察：LMCache 是否能恢复，还是服务降级？

### 节奏

- **每周** — 在预发布环境进行小型金丝雀实验，可能有 5% 生产流量。
- **每月** — 针对特定场景的计划演练日；跨团队参与；事后分析。
- **每季度** — 跨团队弹性审计；依赖关系图更新。

### 工具

- **Harness 混沌工程** — 商业；AI 驱动的实验建议；影响半径缩减；MCP 工具集成。
- **LitmusChaos** — CNCF 毕业项目；基于 Kubernetes 工作流。
- **Chaos Mesh** — CNCF 沙箱项目；Kubernetes 原生 CRD 风格。
- **Gremlin** — 商业；广泛支持。
- **AWS FIS** / **Azure Chaos Studio** — 托管云产品。

### 从小做起

第一个实验：在稳态流量下杀死一个解码副本。观察重路由和恢复。如果这个有效且看起来安全，逐步升级到网络混沌。

第一个 LLM 专属实验：注入一个提供商 429 持续 5 分钟。观察故障转移。大多数团队会发现他们的故障转移没有经过完整测试。

### 需要记住的数字

- 四个平面：控制、目标、安全、可观测性。
- 燃烧率暂停：每日预算消耗 2 倍触发。
- 节奏：每周金丝雀、每月演练、每季度审计。
- 五个 LLM 实验：内存、网络、提供商、畸形提示词、KV 风暴。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| SLI / SLO | "服务目标" | 指标 + 目标；必需的前提条件 |
| 影响半径（Blast radius） | "范围" | 实验影响的服务/用户集合 |
| 燃烧率告警（Burn-rate alert） | "预算卡口" | 错误预算消耗 > 预期 2 倍时触发 |
| 演练日（Game day） | "月度演习" | 计划的跨团队混沌演练 |
| LitmusChaos | "CNCF 工作流" | CNCF 毕业的 Kubernetes 混沌工具 |
| Chaos Mesh | "CNCF CRD" | CNCF 沙箱 Kubernetes 原生混沌工具 |
| Harness CE | "商业 AI 辅助" | 带 AI 建议的 Harness 混沌工程 |
| 畸形提示词（Malformed prompt） | "分词器炸弹" | 使分词卡住的输入 |
| KV 驱逐风暴（KV eviction storm） | "抢占级联" | 大规模驱逐触发重新预填充 |
