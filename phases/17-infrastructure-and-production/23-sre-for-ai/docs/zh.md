# AI 的 SRE——多智能体故障响应、Runbook 与预测性检测

> AI SRE 使用通过 RAG 接地到基础设施数据（日志、Runbook、服务拓扑）的 LLM，来自动化调查、文档化和协调阶段。2026 年的架构模式是多智能体编排——专业智能体（日志、指标、Runbook）由监督者协调；AI 提出假设和查询，人类审批判断性决策。Datadog Bits AI 和 Azure SRE Agent 将此作为托管产品交付。Runbook 正在演进：NeuBird Hawkeye 使用对抗性评估（两个模型分析同一事故；一致 = 高置信，分歧 = 不确定）；操作记忆跨团队变动持久化。自动修复保持谨慎：AI 建议，人类审批。完全自主行动范围很窄（重启 Pod、回滚特定部署），带有严格护栏——任何宣传"一劳永逸"的都是过度承诺。新兴前沿：事故前预测。MIT 研究报告称，在历史日志 + GPU 温度 + API 错误模式上训练的 LLM 提前 10-15 分钟预测了 89% 的故障。预测：到 2026 年底，95% 的企业 LLM 将具备自动故障转移能力。

**类型：** 学习
**编程语言：** Python（标准库，玩具多智能体故障分类模拟器）
**前置知识：** Phase 17 · 13（可观测性）、Phase 17 · 24（混沌工程）
**预计时间：** 约 60 分钟

## 学习目标

- 绘制多智能体 AI SRE 架构图：监督者 + 专业智能体（日志、指标、Runbook）+ 人工审批卡口。
- 解释为什么自动修复范围很窄（重启 Pod、回滚部署）而非宽泛（重构服务）。
- 描述对抗性评估模式（NeuBird Hawkeye）：两个模型达成一致 = 高置信；分歧 = 上报。
- 引用 MIT 89% 早期检测结果以及操作约束：没有执行动作的预测只是仪表板。

## 问题背景

凌晨 3 点，值班工程师被呼叫。"结账服务错误率高。"他们检查 Datadog、Loki、三本 Runbook、部署日志。30 分钟后发现根本原因是 vLLM 因 KV 缓存尖峰导致 OOM。重启 Pod；错误消除。

2026 年，该调查的前 20 分钟是可以自动化的。按服务分组日志、与最近部署关联、匹配 Runbook——全都是 RAG + 工具使用。有监督的智能体可以在人工打开 Datadog 之前完成初步分类并提出假设。

完全自主修复是另一个问题。重启 Pod：安全。扩展 GPU 池：策略允许时安全。重构服务：绝对不行。这门学科的核心是划定这条边界。

## 核心概念

### 多智能体架构

```
          故障事故
             │
             ▼
          监督者
        /    |    \
       ▼     ▼     ▼
  日志智能体  指标智能体  Runbook 智能体
       │     │     │
       └─────┴─────┘
             │
             ▼
        假设 + 证据
             │
             ▼
          人工审批
             │
             ▼
        执行（窄范围动作集）
```

监督者将事故分解为子查询。专业智能体有工具访问权限（日志搜索、PromQL、文档检索）。监督者综合结果，向人工呈现假设 + 证据。人工审批或重定向。

### 自动修复范围

**安全（窄范围）**：重启 Pod、回滚特定部署、在预先审批的范围内扩展池、启用预先审批的功能标志。

**不安全（宽范围）**：更改服务拓扑、修改资源限制、部署新代码、更改 IAM、修改数据库。

任何宣传"一劳永逸"的都是过度承诺。随着 AI SRE 成熟，安全集合会扩大，但边界是真实存在的。

### 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事故。如果它们在根本原因上达成一致，置信度高。如果不一致，将两个假设都展示给人工，由人工升级处理。简单模式，对抗幻觉根本原因的有效过滤器。

### 操作记忆

团队流动是传统 SRE 的无声杀手——部落知识随人员离开。AI SRE 将 Runbook + 事后分析（post-mortem）存储在向量数据库中；智能体在每个新事故中检索。新工程师加入时，AI 拥有完整历史。

### 事故前预测

MIT 2025 年研究：在历史日志、GPU 温度、API 错误模式上训练的 LLM，在测试集上提前 10-15 分钟预测了 89% 的故障。

现实核查：没有执行动作的预测只是仪表板。操作问题是"当我们预测到时，我们做什么？"预防性排水？呼叫器？自动扩缩容？答案因策略而异。

### 2026 年产品

- **Datadog Bits AI** — Datadog 内置的托管 SRE 副驾驶。
- **Azure SRE Agent** — Azure 原生。
- **NeuBird Hawkeye** — 对抗性评估 + 操作记忆。
- **PagerDuty AIOps** — 分类 + 去重。
- **Incident.io Autopilot** — 事故指挥官 + 协调。

### Runbook 即代码

Runbook 从 Confluence 页面演进为带结构化章节（症状、假设、验证、行动）的版本化 Markdown。结构化 Runbook 为 RAG 检索提供更好的输入。任何 AI SRE 推广都应该从将非结构化 Runbook 转化为结构化开始。

### 需要记住的数字

- MIT 早期检测：89% 的故障，10-15 分钟提前量。
- 多智能体分类：监督者 +（日志、指标、Runbook）+ 人工。
- 安全自动修复集合：重启 Pod、回滚部署、在范围内扩展。
- 对抗性评估：两个模型独立；达成一致 = 高置信。

## 练习

1. 运行 `code/main.py`。如果日志和指标智能体不一致，监督者如何解决？
2. 为你的服务定义三个"安全"自动修复动作。为每个提供理由。
3. 编写结构化 Runbook 模板：章节、必填字段、验证命令。
4. 预测性检测提前 12 分钟触发。你的策略是什么——呼叫器、预防性排水，还是两者？
5. 论证一个 3 人团队是否应该在 2026 年采用 AI SRE 还是等待。考虑成熟度、事故量、风险。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| AI SRE | "值班智能体" | LLM 支持的事故调查 + 协调 |
| 监督者智能体（Supervisor agent） | "编排者" | 将事故分解为子查询的顶层智能体 |
| 专业智能体（Specialized agent） | "领域智能体" | 有工具访问权限的子智能体（日志、指标、Runbook） |
| 自动修复（Auto-remediation） | "AI 修复" | 窄范围预审批动作；不是宽泛重构 |
| 操作记忆（Operational memory） | "向量 Runbook" | 向量数据库中的 post-mortem + Runbook |
| 对抗性评估（Adversarial eval） | "双模型检查" | 独立分析；一致 = 高置信 |
| NeuBird Hawkeye | "对抗性那个" | 具有对抗性评估 + 记忆模式的产品 |
| Bits AI | "Datadog 的 SRE 智能体" | Datadog 托管 AI SRE |
| 事故前预测（Pre-incident prediction） | "早期检测" | 故障预测提前 10-15 分钟 |
