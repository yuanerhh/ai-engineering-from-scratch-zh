# 分层架构及其失败模式

> 分层是嵌套的监督者。管理者智能体在子管理者之上，子管理者在工作者之上。CrewAI 的 `Process.hierarchical` 是教科书版本：`manager_llm` 动态委托任务并验证输出。LangGraph 等价物是 `create_supervisor(create_supervisor(...))`。当任务是真实的组织图时，这是自然模式。它也是最有可能崩溃成管理循环的模式——管理者智能体分配工作不当、误解子输出，或无法达成共识。顺序通常胜过它。

**类型：** 学习 + 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 16 · 05（监督者模式）
**预计时间：** 约 60 分钟

## 问题背景

一旦监督者模式清楚了，自然的下一步是"如果工作者本身是监督者会怎样？"团队有子团队；公司有部门的部门。分层架构反映了这一点。

问题：LLM 管理者与人类管理者不同。人类管理者对其下属的了解有稳定的先验。LLM 管理者在每次轮次中从其上下文中的任何内容重新推理组织。该上下文中的微小漂移，整棵树就会错误分配工作。

## 核心概念

### 形状

```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点规划、委托和综合。只有叶子做实际工作。

### 闪光点

- **清晰的组织映射。** 如果真实任务是部门性的（"法务审查文件、财务审查文件、工程审查文件，然后为高管总结"），层次结构是明确的。
- **本地汇总。** 每个子管理者在顶级管理者看到之前综合其团队的输出。顶级管理者看到三个子管理者摘要，不是十五个工作者输出。

### 崩溃点

2026 年事后分析不断发现的三种失败模式：

1. **任务分配错误。** 管理者读取目标，幻觉一个分解，并委托给错误的子管理者。因为子管理者顺从地做被分配的事，错误只在顶级综合时浮现——比人类可以捕获的地方远了一层。
2. **输出误解。** 子管理者返回"无法验证声明 X"。顶级管理者总结为"声明 X 未确认"。含义在每个级别漂移。
3. **共识循环。** 两个子管理者不同意；顶级管理者要求它们协调；它们向下重新委托；工作者重新运行；子管理者返回略有不同的答案；循环。CrewAI 的 `Process.hierarchical` 用步骤限制防范这个，但限制本身现在是一个超参数。

### 决定性问题

顺序（线性管道）与分层：你的任务实际上有独立的子团队，还是一个假装是树的线性流程？如果是后者，使用顺序。如果是前者，使用分层但为明确的协调规则设置预算。

### CrewAI 的实现

`Process.hierarchical` 将管理者 LLM 连接在专业团队之上。管理者：

- 接收顶级任务，
- 将子任务分配给团队，
- 评估团队输出，
- 决定是否接受、重新委托或迭代。

文档：https://docs.crewai.com/en/introduction（在核心概念下查找"分层流程"）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部监督者有自己的图；外部监督者将内部图视为不透明节点。这比 CrewAI 更便于调试（你可以分别单步通过每个图），但更难表达树的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 动手实践

`code/main.py` 运行一个 3 级层次结构：

- 顶级管理者：将任务分为"工程"和"法务"分支，
- 工程子管理者：分为"前端"和"后端"工作者，
- 法务子管理者：一个工作者。

演示将快乐路径（所有人同意）与**扰动路径**进行对比，在扰动路径中顶级管理者的分解将"法务"错误标记为"财务"，并观察错误级联——子管理者顺从地做财务工作，顶级综合者报告财务发现，原始的法务问题没有得到回答。

运行：

```
python3 code/main.py
```

输出显示两条路径，清楚地对比"被要求什么"与"被交付什么"。

## 产出技能

`outputs/skill-hierarchy-fitness.md` 评估给定任务是否应使用分层、顺序或平坦监督者。输入：任务描述、组织结构、协调预算。输出：模式建议和需要防范的具体失败模式。

## 输出

如果你发布分层架构：

- **将树深度限制在 2。** 三层已经对可观察性隐藏了大多数错误。
- **明确的协调预算。** 在顶级管理者必须提交之前设置最大轮数。通常是 2。
- **每次综合的溯源。** 每个节点的摘要必须引用产生它的叶子输出。
- **在分解漂移时发出警报。** 记录管理者每步的分解；与用户查询进行差异比较。如果分解不再涵盖查询，触发警报。

## 练习

1. 运行 `code/main.py` 并比较快乐与扰动路径。需要多少层管理者交接才能让顶级输出完全偏离用户的问题？
2. 添加第三层（顶级 → 子 → 子子 → 工作者）。测量随着深度增长，扰动路径自我纠正与完全偏离的频率。
3. 在每个子管理者实现一个"金丝雀"工作者，总是被问原始用户问题不变。使用金丝雀答案检测分解漂移。当金丝雀与综合答案不同意时，管理者应如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。找出 CrewAI 应用的一个具体护栏（步骤限制、manager_llm 约束）并描述它针对的失败模式。
5. 比较嵌套 LangGraph 监督者与 CrewAI 分层。哪个使协调循环更便宜地检测？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 分层（Hierarchical） | "组织图模式" | 监督者之上的监督者；只有叶子做工作。 |
| 管理者 LLM（Manager LLM） | "老板" | 在内部节点分解、分配和验证的 LLM。 |
| 分解漂移（Decomposition drift） | "老板迷失了方向" | 顶级管理者的拆分不再涵盖原始问题。 |
| 协调循环（Reconciliation loop） | "无休止的会议" | 子管理者不同意；顶级重新委托；工作者重新运行；循环直到预算耗尽。 |
| 深度-2 上限（Depth-2 ceiling） | "不要超过 2 层" | 经验护栏：3+ 层崩溃了可观察性。 |
| 金丝雀问题（Canary question） | "每层的真实依据" | 总是被问原始查询不变的工作者，用于检测漂移。 |
| 溯源链（Provenance chain） | "谁说了什么" | 从每次综合追踪回产生它的叶子输出。 |
