# 角色专业化——规划者、批评者、执行者、验证者

> 2026 年最常见的多智能体分解：一个智能体规划，一个执行，一个批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示词中的 SOP——产品经理、架构师、项目经理、工程师、QA 工程师——遵循 `Code = SOP(Team)`。ChatDev（arXiv:2307.07924）通过带"通信去幻觉"的"聊天链"将设计师、程序员、审查者、测试者链接起来（智能体明确请求缺失的细节）。验证者是承重角色：Cemri 等人（MAST，arXiv:2503.13657）表明每个多智能体失败都可以追溯到缺失或破损的验证。PwC 报告了来自 CrewAI 中结构化验证循环的 7× 准确性增益（10% → 70%）。

**类型：** 学习 + 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 16 · 04（原语模型）、Phase 16 · 05（监督者）
**预计时间：** 约 60 分钟

## 问题背景

通用多智能体系统产生通用输出。群聊中的三个编码者写出三种版本的同样平庸代码。你可以添加更多智能体、更多轮次，仍然无法越过质量阈值。

修复不是更多智能体——而是*不同的*智能体。分配不同的角色。给批评者规划者没有的工具。给验证者一个客观的测试套件。现在系统有内部分歧和有依据的纠正，不只是并行猜测。

## 核心概念

### 四个规范角色

**规划者（Planner）。** 读取目标，产生步骤列表或规范。工具：知识检索、文档。输出：结构化计划。

**执行者（Executor）。** 一次读取一个计划步骤，产生工件。工具：实际工作工具（代码编译器、shell、API 客户端）。输出：工件。

**批评者（Critic）。** 根据规划者的意图审查执行者的输出。工具：对工件的只读访问、静态分析。输出：接受/拒绝并附理由。

**验证者（Verifier）。** 读取工件并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：通过/失败并附证据。

批评者是主观的、有意见的，通常基于 LLM。验证者是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示词：

- **产品经理**写 PRD。
- **架构师**生产系统设计。
- **项目经理**拆分任务。
- **工程师**实现。
- **QA 工程师**运行测试。

每个角色都有严格的输入/输出模式。角色提示词说明角色*是什么*以及它*必须产生什么*。`Code = SOP(Team)` 的表述——确定性 SOP 将一个 LLM 团队变成可预测的管道。

### ChatDev 的通信去幻觉

ChatDev 添加了一个关键动作：当执行者需要计划中没有的特定细节时，它在继续之前明确询问设计者。这防止了 LLM 的经典失败——合理地发明细节。

实现：角色提示词包含"当你需要没有给你的特定信息时，在产生输出之前按名字询问相关角色。"

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 个多智能体执行失败。21.3% 是验证差距——系统在没有人检查的情况下发出了答案。其余 79% 通常追溯到"有一个静默失败或从未运行的检查"。验证是承重角色。

PwC 报告（CrewAI 部署，2025 年）添加结构化验证循环将准确性从 10% 提升到 70%。一个角色带来 7× 增益。

### 批评者与验证者

- 批评者是 LLM，审查工件的质量。主观。可能被合理的散文愚弄。
- 验证者是在工件上运行的确定性程序。客观。给出通过/失败并附证据。

两者都要使用。批评者捕获验证者无法表达的品味问题。验证者捕获批评者看不到的错误，因为它们只在运行时显现。

### 反模式

你系统中的每个角色都是 LLM，每个角色的输出都是"看起来对我来说没问题"。经典的 MAST 失败模式。添加至少一个验证者，其通过/失败由代码而不是 LLM 决定。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)` 是教科书专业化界面。
- **LangGraph** — 节点可以有专业化提示词；边强制执行管道。
- **AutoGen** — 群聊中带单词名称的角色特定 ConversableAgents。
- **OpenAI Agents SDK** — 角色专业化 Agents 之间的交接工具。

## 动手实践

`code/main.py` 实现了一个构建简单 Python 函数的 4 角色管道：

- **规划者**产生规范。
- **执行者**生成代码字符串。
- **批评者**（LLM 模拟）标记明显问题。
- **验证者**在沙盒（`exec`）中针对测试用例运行生成的代码。

演示运行两次：一次执行者产生正确代码（批评者 + 验证者都通过），一次执行者产生不符合规范的代码（批评者错过了错误因为看起来合理，验证者捕获了因为测试失败）。

运行：

```
python3 code/main.py
```

## 产出技能

`outputs/skill-role-designer.md` 接受任务并产生角色名单（3-5 个角色）、每个角色的输入/输出模式，以及验证者检查。在将智能体连接到框架之前使用这个。

## 输出

检查清单：

- **至少一个确定性验证者。** 从不全 LLM。
- **每个角色的显式 I/O 模式。** 规划者返回规范，不是散文；执行者读取该模式。
- **通信去幻觉。** 当信息缺失时执行者必须询问规划者；从不发明它。
- **批评者/验证者排序。** 先运行批评者（便宜，捕获设计问题），然后运行验证者（慢，捕获错误）。
- **循环预算。** 在升级到人类之前最多 2 个批评者-执行者修订轮次。

## 练习

1. 运行 `code/main.py` 并观察验证者如何捕获批评者错过的错误。添加静态分析检查（计算 `return` 出现次数）作为额外的验证者。它捕获了运行时测试错过的什么？
2. 添加第 5 个角色："需求分析师"，将用户愿望翻译为规划者就绪的规范。应该流向它的通信去幻觉请求是什么？
3. 阅读 MetaGPT 第 3 节（"智能体"）。列出 MetaGPT 5 个角色中每个角色的输入/输出模式。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。找出通信去幻觉在哪里打破了否则将是无限的循环。
5. PwC 的 7× 准确性增益来自验证循环。假设三个添加验证者不会有帮助的任务——正确性的确定性检查是不可能的或代价高昂的。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 角色专业化（Role specialization） | "不同智能体，不同工作" | 为规划者/执行者/批评者/验证者角色调整的不同系统提示词。 |
| SOP 模式（SOP pattern） | "编码的标准操作程序" | MetaGPT 的框架：每个角色严格的 I/O 模式将团队变成管道。 |
| 通信去幻觉（Communicative dehallucination） | "询问而不是发明" | ChatDev 模式：当细节缺失时执行者询问规划者而不是编造一个。 |
| 批评者（Critic） | "LLM 审查者" | 主观的、有意见的审查者。捕获品味问题。可能被合理的散文愚弄。 |
| 验证者（Verifier） | "确定性检查" | 基于代码的通过/失败。测试运行器、类型检查器、模式验证器。无法被愚弄。 |
| 验证差距（Verification gap） | "没有人检查" | 21.3% 的 MAST 失败。答案在没有会捕获错误的检查的情况下发出。 |
| 修订循环（Revision loop） | "批评者将其发回" | 批评者拒绝触发带反馈的执行者重新运行。需要预算。 |
| 全 LLM 反模式（All-LLM anti-pattern） | "对我来说看起来没问题" | 每个角色都是 LLM，没有确定性检查。经典的 MAST 失败。 |
