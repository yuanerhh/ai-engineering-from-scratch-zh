# 生产扩展——队列、检查点、持久性

> 将多智能体系统扩展到数千个并发运行需要**持久执行**。LangGraph 的运行时在每个超步之后以 `thread_id` 为键写入检查点（默认使用 Postgres）；工作者崩溃会释放租约，另一个工作者从断点恢复。智能体可以无限期休眠等待人类输入。**MegaAgent**（arXiv:2408.09955）运行了一个带三种状态（空闲/处理/响应）和两层协调（组内聊天 + 组间管理聊天）的每智能体生产者-消费者队列。**Fiber/异步**比每作业一个线程更适合 LLM 流式传输：线程 99% 的时间处于空闲状态等待 token，fiber 在 I/O 上协作式让出。反驳观点：Ashpreet Bedi 的"扩展智能体软件"认为在负载证明之前应该使用 **FastAPI + Postgres + 不额外添加任何东西**——简单的架构比预期走得更远。本课构建一个持久检查点日志，一个带状态转换的每智能体工作队列，一个异步 vs 线程演示，并落实务实的"从简单开始"规则。

**类型：** 学习 + 构建
**编程语言：** Python（标准库，`asyncio`，`sqlite3`）
**前置知识：** Phase 16 · 09（并行群体网络）、Phase 16 · 13（共享内存）
**预计时间：** 约 75 分钟

## 问题背景

一个原型多智能体系统在一台笔记本电脑上以三个智能体在内存事件循环中运行。你迁移到生产环境：

- 智能体有时运行数小时（长时间研究、人机回路等待）。
- 工作进程崩溃。重启会丢失状态。
- 峰值负载是平均负载的 10 倍；你需要水平扩展。
- 用户按智能体运行次数付费；你需要恰好一次的语义来计费。

内存事件循环不能做这些。你需要一个底层的持久执行层。2026 年的规范选项是：

1. 带检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. Actor 模型框架（MegaAgent 的每智能体生产者-消费者）。
4. 手工搭建的 FastAPI + Postgres（Bedi 的论点）。

本课构建每种的缩影。

## 核心概念

### 持久执行，该模式

持久执行引擎在每个"步骤"（LangGraph 语言中的超步）之后持久化完整的程序状态。在崩溃时：

```
工作者在步骤中途崩溃
  -> 租约超时
  -> 另一个工作者接管 thread_id
  -> 从最后一个检查点恢复
  -> 没有重复的副作用
```

使这起作用的要求：

- **可序列化状态。** 所有智能体状态都必须可持久化。带活动数据库连接的函数闭包无法存活。
- **确定性恢复。** 给定相同的状态和相同的输入，智能体产生相同的行动（或为 LLM 调用推迟到外部确定性预言机）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的或使用去重键。

LangGraph 在每个超步后写入检查点；Temporal 在每个活动后写入；Restate 使用事件溯源日志。三者都实现了相同的模式。

### LangGraph 的运行时

每个智能体有一个 `thread_id`；状态是一个类型化字典；每个超步向检查点表写入一行。在恢复时，运行时从最后一个检查点重播，而不是从头开始。智能体可以 `interrupt()` 等待人类输入；运行时持久化并释放工作者。当输入到来时，任何工作者都可以恢复。

这是 2026 年 4 月的参考生产设计。

### MegaAgent 的每智能体队列

arXiv:2408.09955 描述了一个规模实验：一个集群中数千个并发智能体。架构：

```
智能体 i：
  状态 ∈ {空闲, 处理, 响应}
  in_queue   <- 发给智能体 i 的消息
  out_queue  -> 回复 + 副作用

协调者：
  组内聊天  （同一组中的智能体）
  组间管理聊天  （高级路由）
```

两层协调使组内对话密集进行，而组间保持稀疏——这是使数千个智能体的成本保持线性的模式。

### 异步 vs 每作业一个线程

LLM 调用是 I/O 密集型的。等待下一个 token 的线程 99% 的时间处于空闲状态。每个线程消耗约 1MB RAM；在 10,000 个并发调用时，仅堆栈就需要 10GB。

Fiber（Python `asyncio`、Go goroutine、Rust `tokio`）在 I/O 上协作式让出。相同的 10,000 个调用可以舒适地放在一个进程中。在 LLM 智能体规模上，异步不是优化——它是架构。

例外：CPU 密集型后处理（嵌入、分词器技巧）仍然需要线程或进程。将你的 I/O 层与 CPU 层分开。

### Bedi 的反驳

"扩展智能体软件"（Ashpreet Bedi，2026）认为大多数团队在测量负载之前就过度工程化了。务实的默认值：

- FastAPI + Postgres。
- 每次智能体运行是一行；状态用乐观并发原地更新。
- 通过 `pg_notify` 或简单的 Celery 工作者进行后台作业。
- 应用代码中的重试策略。

对于可管理任务上约 100 个并发智能体运行以下的负载，这通常就够了。当你测量到失败时再升级。

规则：当你遇到简单架构无法解决的具体问题时，才采用持久执行框架。过早采用会在不会得到回报的仪式上浪费时间。

### 恰好一次语义

对于付费的智能体运行，你需要"有效恰好一次"（至少一次交付 + 幂等消费者）。工程手段：

- **每次运行的去重键。** 将其包含在每个副作用调用中。
- **发件箱模式。** 副作用首先写入表，然后由单独的进程执行。两步都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，安排补偿。

这些是数据库工程模式，不是 LLM 特定的。LLM 税只是 LLM 调用很慢；其他一切都是标准的分布式系统。

### 彩虹部署

Anthropic 的多智能体研究系统使用"彩虹部署"：多个智能体运行时版本并发运行，这样长时间运行的智能体不必在每次代码部署时被终止。在一部分流量上金丝雀新版本；当旧版本的智能体完成时退役旧版本。

这对于长时间运行的有状态系统来说是标准的；2026 年的适配是智能体可以存活数小时，所以部署周期必须适应。

### 规范的生产检查清单

- 持久状态（检查点、快照或发件箱 + 可重播日志）。
- 幂等副作用。
- LLM 调用的异步 I/O 层。
- 带去重的至少一次交付。
- 有状态工作负载的彩虹/金丝雀部署。
- 可观察性：每智能体追踪、超步审计、重试计数器。

## 动手实践

`code/main.py` 实现了：

- `CheckpointStore` — SQLite 支持的检查点日志，带 thread_id 键。每个超步追加一行。
- `run_with_checkpoint(agent, thread_id)` — 模拟运行中途崩溃；第二个工作者从最后一个检查点恢复。
- `AgentQueue` — 带小型工作队列的每智能体空闲/处理/响应状态机。
- `demo_async_vs_threads()` — 通过 asyncio 和线程运行 500 个并发模拟的"LLM 调用"；报告挂钟时间和峰值内存（近似值）。

运行：

```
python3 code/main.py
```

预期输出：模拟崩溃后检查点恢复成功；异步版本在 < 1 秒内处理 500 个并发调用；线程版本需要几秒钟，并且每个并发单元使用数量级更多的内存。

规范的生产加固：

- **从简单开始（Bedi 规则）。** FastAPI + Postgres，直到你测量到它失败。
- **在优化之前先对所有内容进行监控。** 每次运行延迟直方图，每步时间，重试次数，失败分类。
- **副作用的发件箱模式。** 特别是支付和外部 API 调用。
- **彩虹部署。** 永远不要在部署期间终止正在进行的智能体运行。
- **当**你遇到具体问题时采用持久执行引擎（Temporal/LangGraph/Restate）：数小时的人机回路等待、跨区域协调、复杂的重试/补偿策略。
- **异步用于 I/O 层。** 线程只用于 CPU 密集型后处理。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 持久执行（Durable execution） | "持久化程序状态" | 引擎在每个超步后写入状态；崩溃恢复是确定性的。 |
| 超步（Super-step） | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "智能体运行标识符" | 绑定检查点和恢复逻辑的键。 |
| 幂等性（Idempotency） | "可安全重试" | 重复副作用产生与一次尝试相同的结果。 |
| 发件箱模式（Outbox pattern） | "解耦副作用" | 将意图写入表；单独的执行器执行并标记完成。 |
| 至少一次交付（At-least-once delivery） | "可能重复" | 消息队列语义；去重键使消费者有效一次。 |
| 彩虹部署（Rainbow deploy） | "重叠版本" | 长时间运行工作负载期间多个运行时版本并发。 |
| 异步 fiber | "协作式让出" | 用户模式并发；对于 I/O 密集型负载比线程便宜。 |
| 检查点（Checkpoint） | "状态快照" | 超步边界处的序列化状态；恢复的关键。 |
