# 交接与例程——无状态编排

> OpenAI 的 Swarm（2024 年 10 月）将多智能体编排提炼为两个原语：**例程**（指令 + 工具作为系统提示词）和**交接**（返回另一个 Agent 的工具）。没有状态机，没有分支 DSL——LLM 通过调用正确的交接工具来路由。OpenAI Agents SDK（2025 年 3 月）是生产继任者。Swarm 本身仍然是最干净的概念参考——其完整源代码仅几百行。该模式之所以传播很广，是因为 API 表面大致是"智能体 = 提示词 + 工具；交接 = 返回智能体的函数"。限制：无状态，因此内存是调用者的问题。

**类型：** 学习 + 构建
**编程语言：** Python（标准库）
**前置知识：** Phase 16 · 04（原语模型）
**预计时间：** 约 60 分钟

## 问题背景

每个多智能体框架都想让你学习其 DSL：LangGraph 节点和边、CrewAI 团队和任务、AutoGen GroupChat 和管理者。DSL 是真正的抽象，但它们让这件事感觉比需要的更重。

Swarm 推向相反方向：使用模型已经拥有的工具调用能力。交接变成工具调用。编排者是当前持有对话的任何智能体。状态机隐含在智能体的系统提示词中。

## 核心概念

### 两个原语

**例程（Routine）。** 定义智能体角色和可用工具的系统提示词。将其想象为一组范围化指令："你是一个分流智能体；如果用户询问退款，交接给退款智能体。"

**交接（Handoff）。** 智能体可以调用的一个工具，返回新的 Agent 对象。Swarm 运行时检测 Agent 返回值并切换下一轮的活跃智能体。

这就是整个抽象。

```python
def transfer_to_refunds():
    return refund_agent  # Swarm sees Agent return → switch active agent

triage_agent = Agent(
    name="triage",
    instructions="Route the user to the right specialist.",
    functions=[transfer_to_refunds, transfer_to_sales, transfer_to_support],
)
```

分流智能体的系统提示词使其根据用户消息选择正确的交接。LLM 的工具调用进行路由。

### 为什么它传播很广

- **小 API。** 两个概念可以学习。
- **使用模型已经做的事情。** 工具调用在各提供商之间已经是生产级别的。
- **没有状态机负担。** 你不描述图；智能体的提示词描述他们交接给谁。

### 无状态权衡

Swarm 明确地在运行之间无状态。框架在运行期间保持消息历史，但不持久化任何东西。内存、连续性、长时间运行的任务——都是调用者的问题。

在生产中（OpenAI Agents SDK，2025 年 3 月），这是改变的主要事情之一：SDK 在保持交接原语的同时添加了内置会话管理、护栏和追踪。

### Swarm/交接适合的情况

- **分流模式。** 前线智能体将用户路由到专家。
- **基于技能的交接。** "如果任务需要代码，调用编码者；如果需要研究，调用研究者。"
- **短暂、有界的对话。** 客户支持、常见问题到工单、简单工作流。

### Swarm 困难的情况

- **带共享内存的长会话。** 交接将对话状态重置为新智能体的提示词加历史。没有调用者管理的内存，智能体之间没有持久状态。
- **并行执行。** 交接是一次一个——活跃智能体切换。并行性需要调用者编排多个 Swarm 运行。
- **审计和重放。** 无状态运行很难精确重放；LLM 的交接选择不是确定性的。

### OpenAI Agents SDK（2025 年 3 月）

生产继任者添加了：

- **会话状态。** 跨运行的持久线程。
- **护栏。** 输入/输出验证钩子。
- **追踪。** 每次工具调用和交接都被记录。
- **交接过滤器。** 控制交接时传输的上下文。

交接原语保留；生产人体工程学在其周围被添加。

### Swarm 与 GroupChat

两者都使用 LLM 驱动的路由，但它们在**谁选择下一个**上不同：

- GroupChat：选择器（函数或 LLM）从外部选择下一个发言者。
- Swarm：当前智能体通过调用交接工具选择其继任者。

Swarm 是"智能体决定接下来什么"；GroupChat 是"管理者决定接下来什么"。Swarm 的决定存在于活跃智能体的工具调用中；GroupChat 的决定存在于 `GroupChatManager` 中。

## 动手实践

`code/main.py` 从头实现 Swarm：Agent 数据类、交接机制（工具返回 Agent）和检测智能体切换的运行循环。

演示：分流智能体路由到退款、销售或支持专家。每个专家都有自己的工具。运行循环打印每次交接。

运行：

```
python3 code/main.py
```

## 产出技能

`outputs/skill-handoff-designer.md` 为给定任务设计交接拓扑：哪些智能体存在，它们可以调用哪些交接，传输什么上下文。

## 输出

检查清单：

- **交接日志。** 每次交接写入带发出智能体、接收智能体、上下文快照的追踪事件。
- **上下文传输规则。** 决定交接时移动什么：完整历史（昂贵）、最后 N 条消息或摘要。
- **交接护栏。** 到具有不同工具权限的专家的交接必须经过身份验证——否则提示词注入可以强制不想要的交接。
- **循环检测。** 两个智能体来回交接是常见的失败；用简单的最后 K 环形检查来检测。
- **回退智能体。** 如果交接目标不存在，回退到安全默认值。

## 练习

1. 运行 `code/main.py`，分流到退款智能体。确认第二轮的活跃智能体是退款。
2. 添加循环检测规则：如果相同的两个智能体连续交接了 3 次，强制退出。设计回退。
3. 阅读 OpenAI Agents SDK 文档中的交接过滤器。实现"交接时摘要"版本：传出智能体在传入智能体接管之前将上下文压缩为要点摘要。
4. 比较 Swarm 交接与 GroupChatManager 选择器。哪种模式使提示词注入更严重，为什么？
5. 阅读 Swarm cookbook（https://developers.openai.com/cookbook/examples/orchestrating_agents）。找出 Swarm 做出的一个明确设计决定，OpenAI Agents SDK 改变了或保留了。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 例程（Routine） | "智能体提示词" | 系统提示词 + 工具列表。定义角色和可用交接。 |
| 交接（Handoff） | "转移到另一个智能体" | 活跃智能体可以调用的工具，返回新智能体。运行时切换活跃智能体。 |
| 无状态（Stateless） | "运行之间没有内存" | Swarm 不持久化任何东西；内存是调用者的责任。 |
| 活跃智能体（Active agent） | "现在谁在说话" | 当前持有对话的智能体。交接改变这个。 |
| 上下文传输（Context transfer） | "交接时移动什么" | 传入智能体看到的历史策略：完整、最后 N 条或摘要。 |
| 交接循环（Handoff loop） | "智能体乒乓" | 两个智能体不断互相交接的失败模式。 |
| OpenAI Agents SDK | "生产 Swarm" | 2025 年 3 月后继者；在交接原语之上添加会话、护栏、追踪。 |
| 交接过滤器（Handoff filter） | "传输上的门" | 在交接边界检查和修改上下文的 SDK 特性。 |
