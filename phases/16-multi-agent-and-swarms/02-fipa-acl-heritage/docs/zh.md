# FIPA-ACL 遗产与言语行为

> 在 MCP 之前，在 A2A 之前，有 FIPA-ACL。2000 年，IEEE 智能物理智能体基金会批准了一种智能体通信语言，包含二十个行事语（performatives）、两种内容语言和一组交互协议——合同网、订阅/通知、条件请求。它因本体论开销对 Web 来说太重而在工业界逐渐消失，但 LLM 驱动的多智能体系统复兴正在悄悄地重新实现同样的思想，只是没有正式语义：JSON 合约代替了行事语，自然语言代替了本体论。本课认真阅读 FIPA-ACL，以便你能看到哪些 2026 年的协议决定是再发明，哪些是新颖性，以及当前浪潮将在哪里重新发现 2000 年代已经解决的问题。

**类型：** 学习
**编程语言：** Python（标准库）
**前置知识：** Phase 16 · 01（为什么需要多智能体）
**预计时间：** 约 60 分钟

## 问题背景

2026 年的智能体协议格局很繁忙：工具用 MCP，智能体用 A2A，企业审计用 ACP，去中心化信任用 ANP，自然语言内容用 NLIP，加上 CA-MCP 和二十几个研究提案。每个规范都宣称自己是基础性的。

诚实的解读是，其中大多数都在重新发现一个非常具体的二十年前的决策树。奥斯汀（1962 年）和塞尔（1969 年）的言语行为理论给了我们"话语即行动"。KQML（1993 年）将其转变为一个线协议。FIPA-ACL（2000 年批准）产生了参考标准化：二十个行事语、内容语言 SL0/SL1、合同网和订阅-通知的交互协议。JADE 和 JACK 是 Java 参考平台。这一努力在 2010 年前后消退，因为本体论开销太重，而 Web 正在获胜。

当你看 MCP 的 `tools/call`、A2A 的任务生命周期或 CA-MCP 的共享上下文存储时，你看到的是 FIPA 决策的较软的、JSON 原生的重新实现。了解遗产告诉你两件事：哪些新"创新"实际上是再发明，以及哪些旧的失败模式新规范将重新发现。

## 核心概念

### 言语行为，一段话

奥斯汀注意到有些句子不描述世界——它们改变世界。"我承诺。""我请求。""我宣告。"他称这些为行事话语。塞尔将其正式化为五类：断言、指令、承诺、表达、宣告。KQML（Finin 等人，1993 年）使其对软件智能体可操作：消息是行事语（动作）加内容（动作关于什么）。FIPA-ACL 清理了 KQML 的缺口，并标准化了约二十个行事语。

### 二十个 FIPA 行事语（部分列表）

| 行事语 | 意图 |
|--------|------|
| `inform` | "我告诉你 P 为真" |
| `request` | "我请求你做 X" |
| `query-if` | "P 为真吗？" |
| `query-ref` | "X 的值是什么？" |
| `propose` | "我提议我们做 X" |
| `accept-proposal` | "我接受提议" |
| `reject-proposal` | "我拒绝提议" |
| `agree` | "我同意做 X" |
| `refuse` | "我拒绝做 X" |
| `confirm` | "我确认 P 为真" |
| `disconfirm` | "我否认 P" |
| `not-understood` | "你的消息无法解析" |
| `cfp` | "对 X 的提案征集" |
| `subscribe` | "当 X 改变时通知我" |
| `cancel` | "取消正在进行的 X" |
| `failure` | "我尝试了 X 但失败了" |

完整列表在 `fipa00037.pdf`（FIPA ACL 消息结构）中。重点不是记住它——重点是这些中的每一个都对应于 LLM 协议最终重新添加的一个原语。

### 规范的 FIPA-ACL 消息

```
(inform
  :sender       agent1@platform
  :receiver     agent2@platform
  :content      "((price IBM 83))"
  :language     SL0
  :ontology     finance
  :protocol     fipa-request
  :conversation-id   conv-42
  :reply-with   msg-17
)
```

七个字段承载协议信封；一个字段（`content`）承载有效负载。其余字段正是每次你将重试、线程和本体论附加到 JSON 协议时重新发明的内容。

### 两个遗留平台

**JADE**（Java 智能体开发框架，1999-2020 年代）是最常用的 FIPA 兼容运行时。智能体扩展基类，交换 ACL 消息，在容器内运行，并使用"行为"协调。交互协议库附带合同网、订阅-通知、条件请求和提议-接受。

**JACK**（Agent Oriented Software，商业版）在 FIPA 消息之上强调 BDI（信念-欲望-意图）推理。更正式，采用较少。

一旦 Web 堆栈吞噬了多智能体用例，两者都衰落了。MCP 和 A2A 是 2026 年的运行时"容器"。

### FIPA 消退的原因

- **本体论开销。** FIPA 需要一个共享本体论来解析 `content`。就本体论达成一致是一个多年的标准化过程。Web 只使用了 HTTP + JSON。
- **没有人使用的正式语义。** SL（语义语言）提供了严格的真值条件，但大多数生产系统使用自由格式内容并忽略了形式主义。
- **工具锁定。** JADE 只有 Java；JACK 是商业的。多语言团队绕过了两者。
- **互联网赢得了堆栈。** REST，然后 JSON-RPC，然后 gRPC 替代了 ACL 的传输。

### LLM 复兴是 FIPA-lite

将 FIPA `request` 与 MCP `tools/call` 进行比较：

```
(request                                {
  :sender  agent1                         "jsonrpc": "2.0",
  :receiver tool-server                   "method":  "tools/call",
  :content "(lookup stock IBM)"           "params":  {"name":"lookup_stock",
  :ontology finance                                   "arguments":{"symbol":"IBM"}},
  :conversation-id c42                    "id": 42
)                                        }
```

相同信封，不同语法。两者都携带：谁、对谁、意图、有效负载、关联 ID。两者都不是对另一个的革命——它们是在同一设计上的不同权衡。

Liu 等人 2025 年的调查（"智能体互操作协议调查：MCP、ACP、A2A、ANP"，arXiv:2505.02279）明确了这一谱系：MCP 对应工具使用言语行为，A2A 对应智能体对等言语行为，ACP 对应审计追踪言语行为，ANP 对应去中心化身份扩展。新规范是带 JSON 语法和较松语义的 ACL 后裔。

### 权衡，直白陈述

**FIPA 给你但现代规范放弃的：**

- 正式语义——你可以证明 `inform` 意味着发送者相信内容。
- 行事语的规范目录——你不必重新争论"我们应该有 `cancel` 吗？"。
- 数十年的交互协议模式——合同网、订阅-通知、提议-接受——具有已知的正确性属性。

**现代规范给你但 FIPA 没有的：**

- 与每种现代工具兼容的 JSON 原生有效负载。
- LLM 无需手工编码本体论即可解释的自然语言内容。
- Web 堆栈传输（HTTP、SSE、WebSocket）。
- 通过自描述文档的能力发现（MCP `listTools`，A2A Agent Card）。

为了更容易实现而放宽意图语义。这是确切的权衡。

### 值得移植的交互协议

FIPA 附带约 15 个交互协议。三个值得带入 LLM 多智能体系统：

1. **合同网协议（CNP）。** 管理者发出 `cfp`（提案征集）；投标者回应 `propose`；管理者接受/拒绝。这是规范的任务市场模式（Phase 16 · 16 谈判）。
2. **订阅/通知。** 订阅者发送 `subscribe`；发布者在主题改变时发送 `inform`。这是 2026 年的每个事件总线。
3. **条件请求（Request-When）。** "当条件 Y 满足时做 X。"带前置条件的延迟动作。2026 年的类比是持久工作流引擎中的延迟任务（Phase 16 · 22 生产扩展）。

每个都清晰地映射到现代消息队列、HTTP + 轮询或 SSE 流式传输。

### 放弃本体论时会发生什么

没有共享本体论，智能体从自然语言内容中推断含义。记录在案的 2026 年失败模式是**语义漂移**：两个智能体对稍微不同的概念使用相同的词（`"customer"`），接收方的智能体基于错误的解释行动，没有模式验证器捕获它。FIPA 的本体论要求会在解析时拒绝该消息。

不全面使用本体论的缓解措施：

- `content` 上的 JSON Schema——在线路层拒绝结构错误。
- 类型化工件（A2A）——拒绝错误的模态。
- 信封中的显式行事语——即使内容是自然语言，也使意图明确。

### 2026 年规范，映射到言语行为遗产

| 现代规范 | FIPA 类比 | 保留什么 | 放弃什么 |
|---------|---------|---------|---------|
| MCP `tools/call` | `request` | 显式意图、关联 ID | 正式语义、本体论 |
| MCP `resources/read` | `query-ref` | 显式意图、关联 ID | 正式语义 |
| A2A 任务生命周期 | 合同网 + 条件请求 | 异步生命周期、状态转换 | 正式完整性保证 |
| A2A 流式事件 | 订阅/通知 | 异步推送 | 类型谓词订阅 |
| CA-MCP 共享上下文 | 黑板（Hayes-Roth 1985） | 多写者共享内存 | 逻辑一致性模型 |
| NLIP | 自然语言内容 | LLM 原生 | 模式 |

从上到下读表，模式是：保留结构原语，放弃形式主义，让 LLM 掩盖歧义。

## 动手实践

`code/main.py` 实现了一个纯标准库的 FIPA-ACL 翻译器。它编码和解码规范的 ACL 信封，并展示每个 MCP/A2A 消息形状如何归结为相同的七个字段。演示：

- 将五个 MCP 风格和 A2A 风格的消息编码为 FIPA-ACL。
- 将 FIPA-ACL 解码回现代等价物。
- 使用 `cfp`、`propose`、`accept-proposal`、`reject-proposal` 在一个管理者和三个投标者之间运行玩具合同网谈判。

运行：

```
python3 code/main.py
```

输出是一个并排追踪，显示每个现代消息的 2026 JSON 形式和 FIPA-ACL 形式，然后是合同网投标的往返。相同的协议原语在往返中保留；只有语法不同。

## 产出技能

`outputs/skill-fipa-mapper.md` 是一个读取任何智能体协议规范并产生 FIPA-ACL 映射的技能。在采用新协议之前使用它来回答："这是真正的新鲜事，还是带 JSON 语法的 `inform`？"

## 输出

不要带回 FIPA-ACL。带回它的检查清单：

- 每条消息的意图原语（行事语）是什么？
- 请求-响应和取消是否有关联 ID？
- 是否有显式的内容语言（JSON-RPC、纯文本、结构化类型工件）？
- 交互协议是否是一等公民，还是你从头重新实现合同网？
- 当两个智能体对内容含义不同意时会发生什么（语义漂移）？

在将任何新协议投入生产之前，记录这五个问题的答案。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 言语行为（Speech act） | "做某事的话语" | 奥斯汀/塞尔：话语即行动。ACL 的理论父母。 |
| FIPA | "那个老 XML 东西" | IEEE 智能物理智能体基金会。2000 年标准化了 ACL。 |
| ACL | "智能体通信语言" | FIPA 的信封格式：行事语 + 内容 + 元数据。 |
| 行事语（Performative） | "动词" | 消息的意图类别：`inform`、`request`、`propose`、`cfp` 等。 |
| KQML | "FIPA 的前身" | 知识查询和操纵语言（1993 年）。更简单、更窄。 |
| 本体论（Ontology） | "共享词汇" | 内容语言谈论的概念的正式定义。 |
| SL0 / SL1 | "FIPA 内容语言" | 语义语言级别 0 和 1——正式内容语言家族。 |
| 合同网（Contract Net） | "任务市场" | 管理者发出 cfp；投标者提议；管理者接受。规范的交互协议。 |
| 交互协议（Interaction protocol） | "消息模式" | 具有已知正确性的行事语序列：条件请求、订阅-通知等。 |
