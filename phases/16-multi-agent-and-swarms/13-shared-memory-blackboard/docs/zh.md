# 共享内存与黑板模式

> 2026 年的多智能体系统中并存两种方法：**消息池**（每个人都能看到所有消息，如 AutoGen GroupChat 或 MetaGPT 所用）和**带订阅的黑板**（智能体订阅相关事件，如 Context-Aware MCP 或 Matrix 框架所用）。两者都是多智能体系统中唯一有状态的部分——这意味着两者都是有趣 bug 的所在。参考失败模式是**内存中毒**：一个智能体幻觉出一个"事实"，其他智能体将其视为已验证的事实，准确性以一种比立即崩溃难得多的方式逐渐衰减。本课从标准库构建两种结构，注入一次中毒攻击，并展示在生产中真正有效的三种缓解措施。

**类型：** 学习 + 构建
**编程语言：** Python（标准库，`threading`）
**前置知识：** Phase 16 · 04（原语模型）、Phase 16 · 09（并行群体网络）
**预计时间：** 约 75 分钟

## 问题背景

多智能体系统需要一个地方让智能体共享事实。一个字面上的选项是"在消息中传递所有内容"——但这通过额外的复制重新发明了共享状态。另一个是"给每个人一个全局日志"——但全局日志无限增长并且容易中毒。第三个是"为每个智能体投影一个视图"——可扩展但模式繁重。

当其中一个智能体产生幻觉并将幻觉写入共享状态时，每个读取该状态的下游智能体都将幻觉作为事实采纳。当人类注意到时，推理链已经深入五步，根本原因是曾经写入的第三条消息。调试多智能体准确性衰减比调试崩溃更难。

这就是内存中毒。它是 MAST 分类法（Cemri 等人，arXiv:2503.13657）中记录最多的第二大失败族群，而且是结构性的：任何没有溯源和不可写验证者的共享内存设计最终都会出现这种情况。

## 核心概念

### 两种主要拓扑

**完整消息池。** 每个智能体读取每条消息。AutoGen GroupChat 和 MetaGPT 使用这种方式。简单、透明、可检查，但在约 10 个智能体之后就无法扩展，因为每个智能体的上下文都充满了其他智能体的工作。

```
agent-A ──写──▶ ┌────────────────┐ ◀──读── agent-D
                │   消息池       │
agent-B ──写──▶ │                │ ◀──读── agent-E
                │  （全局日志）  │
agent-C ──写──▶ └────────────────┘ ◀──读── agent-F
```

**带订阅的黑板。** 智能体声明对主题的兴趣；基底只路由相关消息。CA-MCP（arXiv:2601.11595）和 Matrix 去中心化框架（arXiv:2511.21686）使用这种方式。可以扩展得更远，但需要前期模式设计使订阅有意义。

```
                ┌─ 主题: prices ──┐
agent-A ──发布──▶ │                 │ ──▶ agent-D（已订阅）
                 ├─ 主题: orders ──┤
agent-B ──发布──▶ │                 │ ──▶ agent-E（已订阅）
                 ├─ 主题: alerts ──┤
agent-C ──发布──▶ │                 │ ──▶ agent-F（已订阅）
                 └─────────────────┘
```

### 各自的优势场景

- **完整消息池**在智能体数量少（< 10）、异构且对话是短视野时获胜。当每个人都能看到一切时，推断谁说了什么很简单。
- **黑板**在智能体数量多、角色同质但实例众多（群体）且对话是长时间运行时获胜。路由节省了 token 成本和上下文污染。

生产系统通常混合使用：顶层有一个小型完整消息池（规划层），下面是黑板（工作层）。

### 内存中毒场景

三个智能体在处理一项研究任务。智能体 A 是检索智能体。智能体 B 是摘要智能体。智能体 C 是分析智能体。

1. A 获取一个页面并向共享状态写入消息："该研究报告准确性提升了 42%。"
2. 获取的页面实际上写的是"提升了 4.2%"。A 产生了小数点幻觉。
3. B 读取共享状态，写道："报告了大幅 42% 的准确性提升（来源：A）。"
4. C 读取共享状态，写道："建议采用——42% 的提升是变革性的。"
5. 最终报告引用了一个从未存在过的 42% 数字。

没有智能体崩溃。没有测试失败。系统"正常工作"了。幻觉通过共享状态从一个智能体的上下文越过到每个下游智能体的推理中。

### 为什么这是结构性问题

没有共享状态，智能体 A 的幻觉留在 A 的上下文中。下游智能体会重新获取或重新推导，可能会捕获错误。有了朴素的共享状态，A 的上下文变成了所有人的上下文，幻觉被洗白为事实。

问题不在于共享状态本身——而在于**没有溯源且没有独立验证者**的共享状态。三种缓解措施解决了这个问题：

1. **在每次写入时记录溯源。** 共享状态中的每个条目都记录谁写的、何时写的、使用什么提示词，以及（如果适用）智能体引用了什么来源。下游智能体根据溯源带着怀疑态度阅读。
2. **版本化写入；将其视为仅追加。** 更正是一个新条目，取代旧条目，而不是原地更新。审计轨迹得以保留。
3. **保留至少一个无法向共享状态写入的智能体。** 只读验证者智能体对条目进行采样，重新获取来源，并标记不一致之处。因为它无法向池中写入，所以它不会被池中毒。

### 黑板先例（Hayes-Roth，1985）

黑板模式比 LLM 智能体早了四十年。Hayes-Roth（1985 年，"控制的黑板架构"）描述了专家知识源，这些知识源观察一个全局黑板，贡献部分解决方案，并触发其他知识源。2026 年的黑板（CA-MCP、Matrix）是同样的模式，以 LLM 智能体作为知识源，以 JSON 对象作为部分解决方案。旧文献中有关于写竞争、机会控制和一致性的文档化解决方案，这些是现代系统正在重新发现的。

### 投影 vs 完整视图

纯黑板给每个订阅者相同的投影（主题范围）。更激进的设计是**每智能体投影**：每个智能体获得根据其角色定制的视图。LangGraph 的状态缩减器是 2026 年的规范实现——缩减器函数将全局状态折叠为特定角色的切片。

每智能体投影可以扩展得更远，但需要一个模式。没有模式，你就在每个智能体的提示词中重新构建临时投影。

### 写竞争模式

多个智能体同时写入是一个并发问题，不只是 LLM 问题。三种模式有效：

- **顺序写入者（单一生产者）。** 所有写入都通过一个协调智能体进行序列化。简单，但是瓶颈。
- **带版本控制的乐观并发。** 每个条目都有版本；写入者在版本不匹配时失败并重试。经典的数据库技术。
- **主题分区。** 不同的智能体拥有不同的主题。没有跨主题竞争。需要设计分区边界。

大多数 2026 年框架默认采用顺序写入者，因为 LLM 调用足够慢，竞争很少见，瓶颈不会造成伤害。

### 不可写验证者

最承重的缓解措施是只读验证者。实现规则：

- 验证者与团队共享状态（读取黑板或消息池）。
- 验证者没有共享状态的写句柄——只有到独立验证频道的写句柄。
- 验证者独立获取写入中引用的来源。标记不一致。
- 验证者的输出被路由到人类或独立决策智能体，永远不会反馈到池中。

没有这种分离，验证者的输出就会成为池中的新条目，这意味着被中毒的池会中毒验证者，然后中毒其验证结果。

## 动手实践

`code/main.py` 用标准库 Python 实现了两种拓扑，加上一个玩具中毒攻击和三种缓解措施。

- `MessagePool` — 线程安全的仅追加日志，支持完整读出。
- `Blackboard` — 主题键控的发布/订阅，支持每智能体订阅。
- `ProvenanceEntry` — 每次写入都记录（写入者、时间戳、提示词哈希、来源 URI）。
- `PoisoningScenario` — 运行一个三智能体研究任务，其中智能体 A 幻觉出小数点。打印最终报告。
- `Verifier` — 一个重新获取来源并标记不一致的只读智能体。在验证者存在的情况下运行相同场景。

运行：

```
python3 code/main.py
```

预期输出：
- 第 1 次运行（无验证者）：幻觉出的 42% 传播到最终报告。
- 第 2 次运行（有验证者）：验证者标记不一致，池被标记为"已标记"，最终报告包含更正说明。

## 产出技能

`outputs/skill-memory-auditor.md` 是一个技能，用于审计任何多智能体系统的共享内存设计，检查溯源、版本控制和验证者分离。在新的多智能体架构投入生产之前运行它。

## 输出

对于任何共享内存设计：

- 在每次写入时记录溯源：`(写入者, 时间戳, 提示词哈希, 引用的工具调用, 来源 URI)`。
- 使日志仅追加。更正是引用被取代条目的新条目。
- 部署至少一个有独立来源访问权限的只读验证者智能体。
- 将验证者输出路由到独立频道，而不是反馈到共享池中。
- 记录属于取代操作的写入比例——上升的比例是幻觉模式的早期证据。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 消息池（Message pool） | "共享聊天历史" | 每个智能体读取的仅追加日志。完全透明，扩展性差。 |
| 黑板（Blackboard） | "共享工作空间" | 主题键控的发布/订阅。智能体订阅相关主题。扩展性更好。 |
| 溯源（Provenance） | "谁写了什么" | 每次写入的元数据：写入者、时间戳、提示词、来源。 |
| 内存中毒（Memory poisoning） | "幻觉传播" | 一个智能体的错误进入共享状态，下游智能体将其作为事实采纳。 |
| 仅追加（Append-only） | "没有原地更新" | 更正是取代旧条目的新条目。保留审计轨迹。 |
| 不可写验证者（Unwritable verifier） | "独立审计者" | 重新获取来源并标记不一致的只读智能体。 |
| 投影（Projection） | "范围视图" | 从全局状态计算的每智能体视图。LangGraph 缩减器是规范案例。 |
| 知识源（Knowledge Source） | "专家智能体" | Hayes-Roth 1985 年对黑板参与者的术语。 |
