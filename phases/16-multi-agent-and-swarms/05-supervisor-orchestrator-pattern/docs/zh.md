# 监督者/编排者-工作者模式

> 一个主导智能体规划和委托；专业工作者在并行上下文中执行并报告回来。这是 Anthropic 研究系统背后的模式（Claude Opus 4 作为主导，Sonnet 4 作为子智能体），在内部研究评估中比单智能体 Opus 4 提升 +90.2%。Anthropic 的工程文章报告，BrowseComp 上 80% 的方差仅由 token 使用解释——多智能体胜出主要是因为每个子智能体获得全新的上下文窗口。本课从原语构建监督者模式，并涵盖来自生产部署的 2026 年工程经验。

**类型：** 学习 + 构建
**编程语言：** Python（标准库，`threading`）
**前置知识：** Phase 16 · 04（原语模型）
**预计时间：** 约 75 分钟

## 问题背景

研究是单智能体系统失败的典型任务。你问"多智能体系统在 2023 到 2026 年间发生了什么变化？"单个智能体按顺序阅读五篇论文，用其一半的上下文填满它们的文本，然后必须一起推理所有内容。到达第五篇时它忘记了第一篇。它无法并行化。

监督者模式修复了这个问题：一个主导智能体规划搜索，将每个子问题委托给工作者，然后综合。每个工作者为一个窄问题获得自己的 200k token 窗口。主导从不看原始论文——只看工作者摘要。

Anthropic 的生产研究系统报告，与单个 Opus 4 相比，在内部研究评估上提升 +90.2%。同一文章注意到 BrowseComp 上 80% 的方差由*仅 token 使用*解释。每个子智能体的全新上下文是主要机制。

## 核心概念

### 模式

```
                 ┌──────────────┐
                 │   Lead       │  plans, decomposes,
                 │  (Opus 4)    │  synthesizes
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ Worker1 │  │ Worker2 │  │ Worker3 │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         fresh       fresh        fresh
         context     context      context
```

主导从不读原始材料。工作者直到主导综合前从不看彼此的工作。每个箭头是一个带窄工件的交接。

### 为什么它胜出

三个机制：

1. **每个子智能体的全新上下文。** 探索"FIPA-ACL 遗产"的工作者不携带主导花在规划上的 40k token。它为一个问题获得 200k 窗口。
2. **通过提示词的专业化。** 主导的提示词是"分解和综合"，不是"研究"。每个工作者的提示词是窄的："找出 X 中发生了什么变化。"专注的提示词产生专注的输出。
3. **并行性。** 工作者并发运行。挂钟时间大约是 `max(工作者时间) + 规划 + 综合`，不是 `sum(工作者时间)`。

### 工程经验（Anthropic 2025）

Anthropic 文章列出了几个仍然与 2026 年相关的生产经验：

- **根据查询复杂性扩展努力。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。主导必须估计这个，不是调用者。
- **先广后窄。** 先分解为广泛子问题，然后如果答案值得深度则为每个子问题生成更多工作者。
- **彩虹部署。** 智能体是长时间运行且有状态的。传统的蓝绿部署不起作用。Anthropic 使用彩虹：在旧版本耗尽时逐步推出新版本。
- **Token 使用主导。** 多智能体大约是单智能体 token 的 15 倍。只有当任务价值证明成本合理时才运行它。

### LangGraph 的转变

LangGraph 最初附带了一个带高级 `create_supervisor` 助手的 `langgraph-supervisor` 库。2025 年 LangChain 将建议转移到通过工具调用直接实现监督者模式，因为工具调用对*监督者看到什么*（上下文工程）提供更多控制。库仍然有效；文档现在推荐工具调用形式。

### 失败模式

- **主导幻觉规划。** 如果主导生成不分解真实问题的子问题，工作者在错误目标上进行精确研究。
- **工作者过度探索。** 没有明确的范围边界，工作者漂移超出其分配的子问题并污染综合步骤。
- **综合冲突。** 两个工作者返回矛盾的事实。主导必须要么重新询问（添加一轮）要么明确指出分歧。静默选择一方是最坏的失败：用户永远不知道发生了分歧。

### 监督者何时是错误的

- **顺序任务。** 如果步骤 2 实际上需要步骤 1 的输出，并行性没有任何收益。使用管道（CrewAI Sequential、LangGraph 线性图）。
- **简单查询。** 单智能体更快更便宜地处理它们。在生成工作者之前使用主导的"扩展努力"检查。
- **严格确定性。** 监督者使用 LLM 选择的委托。当审计/重放比适应性更重要时，静态图更好。

## 动手实践

`code/main.py` 使用 `threading` 实现了一个三并行工作者的监督者。主导将查询分解为子问题，工作者在每个子问题上并发运行，主导综合。没有真实 LLM——工作者是脚本化的来模拟获取和摘要。

关键结构：

- `Lead.plan(query)` 将查询分为 3 个子问题。
- `Worker.run(sub_q)` 返回一个假摘要（在生产中可以是任何使用工具的智能体）。
- `Lead.run(query)` 在线程中启动工作者，等待完成，并综合。

运行：

```
python3 code/main.py
```

输出显示规划、带开始/结束时间戳的并行工作者追踪，以及最终综合。你可以看到挂钟的收益：三个 0.3 秒的工作者在约 0.35 秒内运行，不是 0.9 秒。

## 产出技能

`outputs/skill-supervisor-designer.md` 接受用户查询并生成监督者模式设计：主导系统提示词、工作者角色、子问题分解规则和综合模板。在构建新的研究风格智能体系统之前使用这个。

## 输出

部署监督者模式前的检查清单：

- **模型配对。** 主导使用推理层模型（Opus 类、`o3` 类）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工作者超时。** 超过 2× 中位运行时间的任何工作者都被终止；主导要么以更窄的范围重新生成，要么在没有它的情况下继续。
- **每工作者 token 上限。** 硬限制（例如预期综合输入的 10 倍）防止失控工作者吹爆预算。
- **可观察性。** 追踪主导的规划、每个工作者的工具调用和综合。这是任何事后调试的基础。
- **彩虹发布。** 有状态的长时间运行智能体需要逐步版本过渡，不是热交换。

## 练习

1. 运行 `code/main.py`，然后修改主导以生成 5 个工作者而不是 3 个。观察挂钟效果。在这个演示中，在什么工作者数量下，生成开销超过并行节省？
2. 实现工作者超时：终止运行超过 0.5 秒的任何工作者，让主导综合剩余结果。你需要什么可观察性来知道工作者被切断了？
3. 在主导的综合中添加冲突检测步骤：如果两个工作者返回矛盾的答案，主导指出分歧而不是选择一个。如何在不调用 LLM 的情况下检测矛盾？
4. 阅读 Anthropic 的研究系统工程文章。列出这个玩具演示需要采用以在生产中运行的三个实践。
5. 比较 LangGraph 的 `create_supervisor`（旧版）与新的工具调用建议。哪个给你更好地控制监督者看到什么？为什么 Anthropic 明确只将子答案而不是原始工作者上下文传入综合？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 监督者（Supervisor） | "主导智能体" | 规划、委托和综合的编排者智能体。不自己做工作。 |
| 工作者（Worker） | "子智能体" | 由监督者以窄范围和自己的上下文窗口调用的专注智能体。 |
| 编排者-工作者（Orchestrator-worker） | "监督者模式" | 相同的东西，不同的名字。2026 年文献两者都用。 |
| 全新上下文（Fresh context） | "干净窗口" | 工作者的上下文从其系统提示词和分配的问题开始，不是主导的历史。 |
| 彩虹部署（Rainbow deployment） | "渐进发布" | 长时间运行的有状态智能体需要版本化的耗尽和替换，不是蓝绿。 |
| Token 主导（Token dominance） | "上下文是变量" | 根据 Anthropic，研究评估方差的 80% 来自使用的总 token，不是模型选择。 |
| 扩展努力（Scale effort） | "将智能体数量与复杂性匹配" | 主导估计查询难度，相应地生成 1 vs 10+ 个工作者。 |
| 综合冲突（Synthesis conflict） | "工作者不同意" | 两个工作者返回矛盾的事实；主导必须表面化分歧，不是静默选择一个。 |
