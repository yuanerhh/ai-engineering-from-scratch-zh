# 失败模式——MAST、群体思维、单一文化、级联错误

> 2026 年的参考分类法是 **MAST**（Cemri 等人，NeurIPS 2025，arXiv:2503.13657），来源于跨 7 个最先进开源 MAS 的 1642 个执行追踪，显示**41-86.7% 的失败率**。三个根本类别：**规范问题**（41.77%）——角色模糊、任务定义不清；**协调失败**（36.94%）——通信中断、状态不同步；**验证差距**（21.30%）——缺少验证、缺乏质量检查。**群体思维**家族（arXiv:2508.05687）补充了：单一文化崩溃（相同基础模型 → 相关失败）、顺从偏见（智能体互相强化错误）、缺乏心智理论、混合动机动态、级联可靠性失败。级联示例：重试风暴，支付失败触发订单重试，触发库存重试，使库存服务不堪重负（几秒内 10 倍负载——需要断路器）。内存中毒：一个智能体的幻觉进入共享内存，下游智能体将其视为事实；准确性逐渐衰减，使根本原因诊断变得痛苦。**STRATUS**（NeurIPS 2025）通过专门的检测/诊断/验证智能体报告 1.5 倍的缓解成功率提升。本课将失败模式视为一等工程目标。

**类型：** 学习
**编程语言：** Python（标准库）
**前置知识：** Phase 16 · 13（共享内存）、Phase 16 · 14（共识与 BFT）、Phase 16 · 15（投票与辩论拓扑）
**预计时间：** 约 75 分钟

## 问题背景

多智能体系统在真实任务上有 41-86.7% 的失败率（Cemri 等人 2025 年跨 7 个开源 MAS 测量了这一点）。这不能通过"只是添加更多智能体"来调试。失败有结构性原因。MAST 分类法给你提供了类别。本课将每个类别映射到具体的检测、诊断和缓解模式，使这些数字不再看起来随意。

2026 年的生产实践是将失败模式视为设计输入。你的架构在你能指出每个 MAST 类别并命名你部署的缓解措施之前，还不够好。

## 核心概念

### MAST 类别

**规范问题（41.77% 的失败）。** 智能体的任务定义不够严格。示例：

- 角色模糊：两个智能体都认为自己是审查者。
- 任务未充分说明："总结这个"，当用户想要特定角度时。
- 成功标准隐含：智能体无法判断是否成功。

缓解措施：
- 编写明确的角色合同。每个智能体的提示词说明它做什么*和什么不做*。
- 每个任务的验收测试。在智能体开始之前，定义"完成看起来像 X"。
- 起飞前规范检查：一个单独的智能体在派发之前审查任务定义。

**协调失败（36.94%）。** 通信或状态中断。

示例：
- 两个智能体在没有同步的情况下更新共享状态。
- 消息在智能体之间丢失（队列失败、超时）。
- 状态漂移：智能体 A 认为任务完成了；智能体 B 仍在执行。

缓解措施：
- 带乐观并发的版本化共享状态。
- 对关键消息的明确确认（重试直到确认）。
- 定期状态同步检查点；及早检测漂移。

**验证差距（21.30%）。** 没有对输出的独立检查。

示例：
- 一个智能体声称成功；没有人验证。
- 智能体链中每个都信任前一个的输出。
- 涌现的组合行为缺少测试覆盖。

缓解措施：
- 独立验证者智能体（第 13 课）。只读，独立来源访问。
- 明确的交接合同："A 的输出在 B 开始之前必须通过检查器 C。"
- 结果记录用于事后分析。

### 群体思维家族（arXiv:2508.05687）

当智能体同质化或互相模仿时的五种相关失败：

**单一文化崩溃。** 相同的基础模型或训练数据 → 相关错误。当三个智能体共享一个 LLM 时，它们共享其幻觉。

**顺从偏见。** 智能体向最大声或最自信的同伴调整，即使是错的。

**缺乏 ToM。** 智能体未能对彼此的信念建模；协调瓦解（第 18 课）。

**混合动机动态。** 具有部分一致激励的智能体向妥协中间值漂移，这让所有人都不满意。

**级联可靠性失败。** 一个组件的错误模式触发依赖组件中的错误模式。

### 级联示例——重试风暴

一个经典的 2026 事故模式：

```
支付服务失败 10% 的请求
   ↓
订单智能体重试支付（指数退避但很朴素）
   ↓
每次重试都是一次新的订单-库存检查
   ↓
库存服务看到正常负载的 2 倍
   ↓
库存服务开始超时
   ↓
每个订单重试库存检查
   ↓
库存服务看到正常负载的 10 倍
   ↓
集群崩溃
```

解决方案是经典的：**断路器**。当下游错误率超过阈值时，用缓存或默认结果短路。加上每个请求的限制重试预算。

断路器是你直接从分布式系统借用而无需修改的少数几种多智能体失败缓解措施之一。

### 内存中毒（重温）

来自第 13 课：一个智能体的幻觉成为共享内存事实；下游智能体基于中毒的事实推理。在 MAST 术语中，这是共享内存层的验证差距。

渐进的准确性衰减是症状。你不会得到崩溃；你会得到缓慢的漂移，难以找到根本原因。

缓解措施：仅追加日志、溯源、不可写验证者。已在第 13 课中涵盖。

### STRATUS——用于失败检测的专门智能体

STRATUS（NeurIPS 2025）在你部署以下智能体时报告 1.5 倍的缓解成功率提升：

- **检测智能体。** 监视症状模式（高度分歧、重试峰值、准确性漂移）。
- **诊断智能体。** 给定症状，从 MAST 分类法推断可能的根本原因。
- **验证智能体。** 在应用缓解措施后，检查症状是否消除。

这是 SRE 风格的事故响应，应用于智能体系统。三个角色都可以是带专门提示词的 LLM 智能体。

### 失败模式审计

2026 年的最佳实践是每年（或每个主要版本）进行一次失败模式审计：

1. **追踪采样。** 收集约 1000 个真实执行追踪。
2. **分类。** 对于每个追踪的失败，映射到 MAST + 群体思维类别。
3. **按类别计算失败率。** 哪些类别在你的系统中占主导？
4. **排列缓解措施。** 哪个修复会消除最多的失败？
5. **选择 2-3 个缓解措施。** 实施；下一季度重新审计。

纪律比具体的选择更重要。没有审计，失败会融入噪音，永远不会被系统性地处理。

### 当系统悄悄失败时

最危险的失败类别是无声的正确性失败。大声失败的系统（崩溃、异常、警报）可以被监控。产生合理但错误输出的系统不能被异常日志检测到。这就是为什么验证差距是每次失败成本最高的类别，即使按数量只有 21.30%。

投资于：
- 基于采样的人工审查。
- 黄金数据集回归测试。
- 对重要输出的跨智能体交叉检查。

### 失败 vs 慢失败

有些失败是立即的；有些是缓慢的。立即失败（超时、模式不匹配、认证错误）检测成本低。慢失败（内存中毒、单一文化漂移、角色模糊）检测和预防成本高。

2026 年的工程手段：监控慢失败代理指标，这样你可以在漂移成为可见错误之前捕获它。一致率、重试率、输出长度分布和连续智能体版本之间的编辑距离都是有用的代理指标。

## 动手实践

`code/main.py` 实现了：

- `FailureTaxonomy` — 将模拟事故分类为 MAST + 群体思维类别。
- `CircuitBreaker` — 经典模式；当错误率超过阈值时打开。
- `RetryStormSimulator` — 展示级联失败；开/关断路器切换。
- `DetectionAgent` — 脚本化的 STRATUS 风格症状匹配器。

运行：

```
python3 code/main.py
```

预期输出：
- 没有断路器的重试风暴：库存错误爆炸（模拟）。
- 有断路器：在阈值处限制；提供降级模式响应。
- 检测智能体标记该模式并命名 MAST 类别。

## 产出技能

`outputs/skill-mast-auditor.md` 对多智能体系统运行 MAST 风格的失败模式审计。追踪 → 分类 → 缓解措施排名。

## 输出

生产中的失败模式纪律：

- **每季度进行 MAST 审计。** 不是每年。随着你的系统增长，类别会改变。
- **到处部署断路器。** 对任何依赖服务的每个出站调用。默认在 5-10% 错误率时打开。
- **黄金数据集。** 小型、高质量、手工审计。每周进行回归测试。
- **STRATUS 三重奏。** 监控生产的检测 + 诊断 + 验证智能体。只从检测智能体开始；当症状嘈杂时再添加诊断。
- **失败预算。** 按类别的显式 SLO 失败率。超出预算触发停止发布对话。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| MAST | "2026 年分类法" | Cemri 2025；3 个根本类别 + 14 种失败子类型。 |
| 规范问题（Specification Problem） | "角色模糊" | 任务或角色定义不足；智能体不知道该做什么。 |
| 协调失败（Coordination Failure） | "状态漂移" | 智能体之间的通信或同步中断。 |
| 验证差距（Verification Gap） | "没有人检查" | 输出在没有独立验证的情况下被接受。 |
| 群体思维家族（Groupthink family） | "同质化失败" | 单一文化、顺从、缺乏 ToM、混合动机、级联。 |
| 单一文化崩溃（Monoculture collapse） | "相同模型，相同幻觉" | 来自共享基础模型或训练数据的相关错误。 |
| 重试风暴（Retry storm） | "级联错误放大" | 一个失败触发重试，这会在下游放大负载。 |
| 断路器（Circuit breaker） | "在错误率上快速失败" | 当错误率超过阈值时打开；用默认值短路。 |
| STRATUS | "事故响应三重奏" | 检测 + 诊断 + 验证智能体。1.5 倍缓解成功率。 |
| 内存中毒（Memory poisoning） | "幻觉传播" | 共享内存事实被污染；下游智能体基于中毒推理。 |
