# 并行/群体/网络化架构

> 与监督者形成对比：没有中央决策者。智能体读取共享事件总线，异步接取工作，写回结果。LangGraph 明确支持去中心化动态环境的"群体架构"。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，以消除编排者瓶颈。权衡是明确的：为了可扩展性而牺牲确定性和可追踪性。群体适合有许多独立子问题的任务；它不适合需要单一连贯计划的任务。

**类型：** 学习 + 构建
**编程语言：** Python（标准库，`threading`，`queue`）
**前置知识：** Phase 16 · 05（监督者模式）、Phase 16 · 04（原语模型）
**预计时间：** 约 75 分钟

## 问题背景

监督者可以扩展到几个工作者。那数百个呢？监督者本身成为瓶颈：每个关于谁做什么的决策都通过一个智能体进行。一个缓慢的规划步骤会停滞整个系统。

群体架构翻转了设计。不是一个中央规划者分配工作，而是工作者从共享队列中接取工作。"协调"烘焙在事件总线语义中。没有编排者；系统扩展直到队列为止。

## 核心概念

### 形状

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

没有编排者。每个工作者重复：拉取任务、处理、写结果（可选地将后续任务入队）。

### 群体适合的情况

- **许多独立任务。** 抓取、转换、分类。任务彼此不依赖。
- **可变持续时间的工作。** 如果某些任务需要 100ms，而其他需要 10 秒，群体自动平衡负载——快速工作者拉取下一个工作。监督者必须预测持续时间。
- **吞吐量优于确定性。** 你关心总完成时间，不是严格排序。

### 群体失败的情况

- **有序工作流。** 如果步骤 3 需要步骤 2 的输出，群体有在步骤 2 完成之前触发步骤 3 的风险。
- **全局规划任务。** 复杂研究问题受益于规划者。研究者群体产生独立事实，不是连贯报告。
- **调试。** 没有中央日志和异步工作，重现错误代价高昂。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年将群体带到其自然结论的论文：控制流和数据流都是分布式队列上的序列化消息。没有中央协调者。容错来自消息持久性。可扩展性是消息代理的问题，不是系统的。

贡献：一个编程模型，其中多智能体协调是"这个智能体订阅什么消息主题？"而不是"监督者下一步选择哪个智能体？"这使系统看起来像一个 pub/sub 事件网格。

### LangGraph 的群体架构

LangGraph 2025 文档明确将"群体架构"描述为多智能体模式之一：智能体是节点，但边形成带循环的有向图，任何节点都可以从池中被激活。工作者根据条件从可用工作中拉取，不是根据监督者分配。

### 失败模式：饥饿和热点

如果所有工作者拉取最快可用的任务，长时间运行的任务永远不会被选取，直到它们是唯一剩下的。经典的队列饥饿。

缓解措施：
- 带明确老化的优先级队列（随等待时间增加优先级）。
- 工作者专业化：一些工作者只接受"长"任务。
- 背压：限制进入队列的快速任务数量。

### 内容路由链接

群体与内容路由（第 22 课）自然配对。不是通用队列，而是每种消息类型一个队列。专业工作者只订阅其类型。这是扩展到数千个智能体的消息总线架构的基础。

## 动手实践

`code/main.py` 实现了一个从共享 `queue.Queue` 拉取的 4 个工作者线程的群体。任务有可变持续时间（有些快，有些慢）。演示对比：

- **顺序基线：** 一个工作者串行处理所有任务。
- **固定分配：** 每个任务预先分配给特定工作者（监督者风格）。
- **群体：** 工作者从共享队列拉取。

群体自动平衡负载；固定分配在其分配的任务缓慢时让快速工作者空闲。

运行：

```
python3 code/main.py
```

输出显示每个工作者的任务数量（群体不均匀但最优地分配）和挂钟时间。

## 产出技能

`outputs/skill-swarm-fit.md` 评估任务是否应使用群体与监督者。输入：任务独立性、持续时间方差、排序要求、可调试性需求。

## 输出

检查清单：

- **带老化的优先级队列。** 防止长任务饥饿。
- **工作者幂等性。** 如果工作者在运行中途崩溃，任务可能被拉取多次。工作者必须是幂等的。
- **持久队列。** 生产中使用 Kafka、Redis Streams 或数据库支持的队列。`queue.Queue` 只在内存中。
- **每任务可观察性。** 每个任务有追踪 ID；每个工作者用它记录开始/结束。
- **背压。** 如果队列增长比工作者耗尽更快，减慢生产者。

## 练习

1. 运行 `code/main.py`。群体比顺序处理可变持续时间工作负载快多少？比固定分配快多少？
2. 添加优先级队列变体（使用 `queue.PriorityQueue`）。按任务"重要性"字段分配优先级。观察在持续负载下低优先级任务是否会饥饿。
3. 实现热点检测器：记录任何工作者处理的任务比最慢工作者多 3 倍。这表明任务持续时间分布有什么问题？
4. 阅读 Matrix 论文（arXiv:2511.21686）摘要和第 3 节。找出 Matrix 接受的一个具体权衡（可扩展性增益）和它放弃的一个（可追踪性、确定性）。
5. 将群体演示转换为使用（任务类型、有效负载）元组的 `queue.Queue`，工作者只订阅特定类型。当任务是异构的时，哪些路由规则有意义？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| 群体架构（Swarm architecture） | "去中心化智能体" | 工作者从共享队列拉取；没有中央编排者。 |
| 事件总线（Event bus） | "智能体订阅主题" | 按类型或内容将任务路由给工作者的消息代理。 |
| 饥饿（Starvation） | "任务永远不运行" | 低优先级任务永远不被选取，因为高优先级工作持续到来。 |
| 热点（Hot-spotting） | "一个工作者溺水" | 负载不平衡，一个工作者获得大多数任务。 |
| 背压（Back-pressure） | "减慢生产者" | 当队列填满时向上游发出停止生产的信号的机制。 |
| 幂等工作者（Idempotent worker） | "可安全重新运行" | 处理两次的任务产生相同的结果。必需的，因为工作者可能在运行中途崩溃。 |
| 持久队列（Durable queue） | "在崩溃中存活" | 由磁盘或复制存储支持的队列；当工作者崩溃时任务不会丢失。 |
| Matrix 框架（Matrix framework） | "全消息传递群体" | 数据流和控制流都是分布式队列上的序列化消息。 |
